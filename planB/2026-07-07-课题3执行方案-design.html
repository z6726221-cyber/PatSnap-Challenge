课题3 执行方案 · Plan B（LLM Wiki 路线）

审阅版 · 与 planB/2026-07-07-课题3执行方案-design.md 完全对齐
和 Plan A 的关系：Plan A（planA/）是纯向量数据库路线，Plan B（本文档）是 LLM-Wiki-Compiled RAG 路线。两者是并列的技术路线沙盘，不是新旧替代——本文档是 Plan B 分支的权威版本，此前 docs/superpowers/specs/ 下的同名文件已归档为历史快照，不再同步更新。

0. 项目定位与约束

0.1 一句话定位

智慧芽的运营、销售不懂技术（TRIZ、专利分析算法、AI 能力……），但天天要面对客户和市场，被追问技术问题。过去只能问研发、翻文档，慢且答案不统一。本产品把智慧芽的技术资料沉淀成一个 AI 可调用的知识库，让运营/销售当场问、当场答、口径统一，也能当场拿竞品网址做对比，说清楚自己的优势。

0.2 用户故事（贯穿汇报的叙事主线）

小明是智慧芽的运营同学，不懂 TRIZ（这只是一个例子，换成任何一个智慧芽的技术领域都一样）。过去他要面对：

  • 客户问"TRIZ 和专利分析有什么关系？"，答不上来，只能说"我问一下研发"——慢，也显得不专业。
  • 同事老王上周跟客户说的技术优势，和小明这周说的不一样——客户当场就能听出口径不一致，信任度直接掉一截。
  • 销售在客户面前被甩了一个竞品的官网链接，"你们比它强在哪？"——现场答不出来，只能事后补，已经错过了成单的窗口。

现在，小明打开知识库，问一句就有准确、统一的答案，附带来源；甩一个竞品网址进去，几分钟内拿到有理有据的对比。这就是本产品要交付的东西。

0.3 核心痛点（按严重程度排序，决定资源投入优先级）

  1. 技术口径不一致〔最严重〕：不同销售/运营对同一技术点的说法不同，客户面前容易露馅，直接损害可信度。
  2. AI 味 / 不像智慧芽自己的风格：生成内容和企业长期形成的表达习惯不一致（注：这条只在"生成宣传材料"类任务里生效，技术问答类任务不追求"官宣腔"，见 §1.5）。
  3. 术语易错：专业术语、产品命名混淆，答错会被内行客户当场揪出来。
  4. 依赖研发响应慢：运营/销售的技术问题要转研发解答，沟通成本高。
  5. 资料分散未被系统化利用：内部文档、论文、项目经验散落各处，没人整理成能被 AI 直接用的形态。
  6. 竞品分析和技术路线判断效率低：靠人工检索归纳，慢。

资源投入原则：优先打穿 #1（统一口径）和 #6 里的"竞品临场对比"——这是运营/销售的日常真实场景，也是这个产品和"一个普通聊天机器人/普通RAG"最大的差异点。#2 风格漂移退居次要，只在生成宣传文案时处理。

0.4 课题原文对照（评分依据，不可丢）

课题原文任务对应本 spec 能力

范围原则（窄而深，不铺开）：知识库深度覆盖 1-2 个技术领域（如 TRIZ，示例不限于此），不追求覆盖智慧芽全部技术资产。"出彩"来自这 1-2 个领域被问穿问透都答得准、答得一致，不来自语料条目数量——多做几个领域只会把策展精力摊薄，现场深挖细节反而露怯。可扩展到更多领域是"未来规划"里讲的故事，不是本次要交付的工作量。

0.5 核心架构：Wiki-Compiled RAG

一句话定调：不是 Wiki 替代 RAG，也不是 RAG 加个 Wiki 壳，而是"编译后的 RAG"——先把散落的原始资料离线编译成结构化的企业记忆 Wiki，运行时再用混合检索召回 Wiki 页面（整页，非碎片），最后经 Skill 工作流生成业务产出。

这从第一性原理推出，五个基本事实决定了它：

  1. 模型不知道企业私域知识 → 必须外部注入。
  2. 上下文窗口有限 → 不能全塞，必须检索。
  3. 原始资料散、旧、冲突、会更新 → 直接 RAG 会召回脏知识/过期知识，检索前要先"编译"，且要能区分新旧（见 §1.2 时间线管理）。
  4. 企业任务要的不只是事实，还有统一的口径——同一个问题，不管谁问、怎么问，都要落到同一个"标准答案"上。这是纯向量库检索做不到的：两次问法不同，召回的碎片可能不同，口径依旧不一致。Wiki 页面是唯一标准答案，不管怎么问、查到就是同一页，这是"统一口径"这个最高优先级痛点的直接解法。
  5. 企业答案必须可追溯 → 不能写得漂亮却不知依据在哪，客户追问一句"这个数据哪来的"要答得上来。

由此得到五层架构（本 spec §1 逐层展开）：

┌─ 1.1 Raw Sources    原始证据层（只读、不可变，事实底座）
├─ 1.2 LLM Wiki       企业理解层（编译后的结构化页面 + index + graph + 时间线）
├─ 1.3 SOUL           企业人格层（术语 + 风格，术语恒定注入，风格按任务场景加载）
├─ 1.4 Retrieval      混合检索层（意图路由 → 整页加载 → 竞品临场抓取）
├─ 1.5 Generation     生成与自检（按任务场景区分轻重）
└─ 1.6 Governance     治理层（受控生长、写回、Lint、版本冲突裁决）

关键分工（贯穿全文）：

  • RAG 解决"运行时去哪找"，Wiki 解决"平时怎么存、怎么保证问出来的答案永远一致"。
  • Raw Sources 永远是事实底座，Wiki 只是编译层——每个结论都能从 Wiki 页回查到原始资料。
  • 确定性工作交给脚本（解析、建索引、扫描建图、校验），认知工作交给 Claude（编译页面、策展、生成、判断）。

0.6 关键约束（写死，防跑偏）

  • 知识组织 = LLM Wiki：资料编译成互链 Markdown 页面，agent 通过 index → 读页 → follow-link 使用。
  • 检索 = Wiki-Compiled RAG：向量/关键词只用于"定位到哪一页"，命中后加载整页（见 §1.4）。
  • 不引入 SAG / 重型知识图谱：本产品的问答、竞品对比、路线判断都没有"因果链多跳追根因"的需求，只需轻量 graph.json 做够用的关系扩展。理由见 §1.2。
  • 知识库范围窄而深：1-2 个技术领域，见 §0.4。
  • 风格分场景：技术问答类任务追求"通俗准确"，不套官宣腔；宣传材料生成类任务才套完整 SOUL 风格自检。见 §1.5。
  • 竞品临场对比支持真实时联网抓取：这是产品的核心能力之一（回应销售被临场甩竞品链接的真实场景），不是"准备阶段预置结果"。见 §1.4、§4.1。
  • 资料会更新，必须能区分新旧：Wiki 页面带变更记录，旧结论被取代时要能追溯，见 §1.2。
  • 可溯源：每个事实点绑回来源；无来源支撑标"待核实"，不编造。

0.7 开工前必须先确认的输入（缺则先问，不许瞎猜）

  • 技术领域清单：本次深度覆盖哪 1-2 个技术领域（TRIZ 只是示例，需用户点名实际范围）
  • 语料清单：该领域的内部文档、论文、历史宣传材料、开源项目介绍的实际文件或 URL
  • 示例竞品：至少 1 个真实竞品网站 URL，用于验证"临场抓取对比"链路
  • Smart-Doc：BASE_URL 与调用凭证/鉴权方式
  • 大模型 API：生成用的 key 与模型名
  • 向量库：选型、连接信息
  • 术语真值来源：可核对智慧芽标准术语的官网/产品界面/论文入口

1. 五层引擎（各产出共享的底层）

1.1 Raw Sources — 原始证据层

定位：只读、不可变的事实底座。Claude 只从这里读，绝不改写。

tools/parse_docs.py — Smart-Doc 适配器

  • 三端点：/image/smart-doc、/pdf/smart-doc、/document/smart-doc（Office 先转 PDF）。
  • 请求参数：detailed:true, markdown:true, ocr_backend:"auto", sign_url:false, reject_aspect_ratio:false；PDF 加 pdf_render_dpi:200, pdf_page_parallelism:4。
  • 响应流式字符串块，最后一块是拼接好的 Markdown → 落地 raw/<doc_id>.md。
  • 必须本地缓存：文件内容 hash 为 key。

Claude 做的：读 raw/ → 去噪 → 切分 → 判定风格样本 → 补元数据，含资料日期（用于 §1.2 时间线管理）。

切片元数据结构：

{
  "chunk_id": "triz-basics-0007",
  "parent_doc_id": "triz-basics",
  "source_type": "domain-doc|paper|official|competitor",
  "domain": "triz",
  "usage": ["技术问答", "宣传材料"],
  "title": "TRIZ 技术矛盾基础",
  "doc_date": "2025-03",
  "source_url": "https://...",
  "content": "……"
}

corpus_manifest.json（根目录，Claude 手动维护）：

[
  {"file": "corpus/domain-docs/triz-intro.md", "type": "domain-doc", "domain": "triz", "doc_date": "2025-03", "usage": ["技术问答"]}
]

验收：给定一份 PDF/文档，跑 parse_docs.py 得到 raw/ 下干净 Markdown（二次调用命中缓存）；Claude 能据此产出带完整元数据（含日期）的切片。

1.2 LLM Wiki — 企业理解层

定位：把 Raw Sources 编译成结构化、互链、可溯源的 Markdown 页面。它是编译层，不是事实源——事实源永远是 raw/。

为什么是"编译"而不是"堆文档"：直接 RAG 每次从原始碎片现找现拼，不改变知识本身的散/旧/冲突状态，也无法保证"两次问法不同、答案还一致"。LLM Wiki 在检索前加一道编译：把散落矛盾的材料预加工成唯一标准答案页，这正是解决"口径不一致"（§0.3 最高优先级痛点）的机制，也是本方案区别于普通 RAG 的根本。

页面类型：

  • domain/ 目标技术领域的知识页（如 domain/triz/技术矛盾.md）——这是知识库主体，投入最多策展精力
  • product-capability/ 智慧芽相关产品能力页
  • competitors/ 每个竞品一页（含历次临场对比的沉淀记录）
  • roadmaps/ 技术路线专题页

页面模板（frontmatter 承载结构化关系字段与时间线）：

---
title: 技术矛盾（Technical Contradiction）
type: domain                # domain|product-capability|competitor|roadmap
status: confirmed            # draft|confirmed
confidence: high             # high|medium|low
sources:
  - {ref: triz-basics, url: "https://...", doc_date: "2025-03"}
related:
  - product-capability/专利矛盾检索
superseded_by: null           # 若本页已被更新页面取代，填新页路径；否则 null
updated: 2026-07-07
---

## 概述
（正文，每个事实点后标 [来源: parent_doc_id, 2025-03]）

## 变更记录
- 2026-07-07：初版，来源 triz-basics（2025-03 文档）

## 待核实
- （不确定的内容放这里，不写进正文当事实）
时间线/版本管理（新增）：每条 sources 带 doc_date。多源冲突时优先信更新日期更近的资料，但如果旧资料是官网/权威定义、新资料是内部草稿，仍按 §1.6 权威性规则裁决，不是唯写"越新越对"。页面若被后续资料证明过时，不直接覆盖删除：新建或更新页面，旧页面填 superseded_by 指向新页面，变更记录 段留痕新旧依据。检索时默认只返回未被指向的最新页；旧页仍可被回溯查阅。lint_wiki.py 新增检查项：长期未更新但所属领域已有更新资料入库的页面，标记"可能过期，待复核"。

两个特殊导航文件：

  • index.md — 全局目录，头部 aliases 表，下方分类列出每页+摘要。
  • graph.json — 轻量关系图：
{
  "nodes": [{"id": "domain/triz/技术矛盾", "type": "domain", "title": "技术矛盾"}],
  "edges": [{"from": "domain/triz/技术矛盾", "to": "product-capability/专利矛盾检索", "source": "frontmatter.related"}]
}

为什么用 graph.json 而非 SAG/知识图谱：不需要"因果链多跳追根因"的能力（技术问答要单点准确答案、竞品对比横向填矩阵、路线判断做能力盘点），graph.json 沿边走一两跳就够。

策展指令（Claude 执行）：

  1. 读 §1.1 产出的切片与整篇文档。
  2. 每个知识点/实体建一页，用模板填写；related 写清结构化关系；sources 带 doc_date。
  3. 事实点标 [来源]；不确定的进 待核实 段和 error-book。
  4. 建 [[双向链接]]。
  5. 新建页 status: draft，人工确认后改 confirmed。
  6. 页面落盘后跑 build_index.py 和 build_graph.py。

验收：目标技术领域的 Wiki 页面能覆盖运营/销售常见追问角度（用一批真实追问样例验证，见 §5）；index.md 覆盖全部页面；graph.json 边合法；每个事实有来源和日期、可回查 raw/；存在版本更新时 superseded_by 链条正确。

1.4 Retrieval — 混合检索层（含竞品临场抓取）

核心思想：检索只负责定位"读哪几页"，命中后加载整页 Wiki（非碎片）。

检索按意图路由：

意图识别（Claude 判断）
├─ 精确查询：用户输入包含与 index.md 里某页 title 高度匹配的实体/概念名
│    → 读 index.md 定位 → 直接加载该页整页 + related 页
│    → 不走向量库  ← 技术问答的常见形态
│    fallback（点名了但查不到）：退化为模糊查询，输出注明"未精确定位，以下是最相关召回"
│
├─ 关系查询：概念名 + 关联词（"和…有关的"、"上下游"）
│    → 沿 graph.json 边遍历（一两跳）→ 加载相关页整页
│
└─ 模糊查询：无法归入以上两类的关键词/问句
     → index.md 收窄范围 → 混合检索（BM25+向量，RRF融合）粗筛候选页 → 整页加载
竞品临场抓取（新增，§4.1 核心能力）： 
用户输入一个竞品 URL
→ tools/fetch_competitor.py 实时请求该 URL，解析为干净文本（同 Smart-Doc 思路，轻量版）
→ 与本地 wiki/competitors/ 已有该竞品页（若存在）合并；若无该竞品页，本次抓取内容直接作为临时素材，不建正式页
→ 与 wiki/product-capability/ 相关页做对比 → 生成对比结论
→ 高价值的对比结论可选择性写回 wiki/competitors/（用户确认后），沉淀为下次可复用的知识
  • 异常处理：URL 打不开/内容抓取失败/反爬限制 → 明确告知用户"抓取失败，原因：xxx"，不编造竞品信息。这是真实的网络请求，会失败，必须有清晰的失败提示，不能静默假装成功。
  • 和"不做现场实时联网"的关系：这条约束原指"AI 技术趋势/竞品资料的准备阶段调研"不必现场演示时才查，但销售被临场甩竞品链接是产品的核心使用场景，必须支持真实时抓取。两者不矛盾：一个是"准备阶段该不该现场查"，一个是"产品本身该不该具备临场查的能力"——后者是必须有的产品功能。

向量库的定位：知识库范围窄而深（1-2 领域），index.md + grep 基础上足够；向量库仍建，供模糊查询和未来扩展领域用。embedding 模型：中英双语（如 BGE-M3 类）。

1.5 Generation — 生成与自检回路（按任务场景区分轻重）

这是本次修订的关键改动：不同任务场景，加载的东西不同，自检回路轻重不同。

任务场景加载什么自检回路
技术问答（tech-qa.md，主 Demo）术语（恒定）+ 整页素材术语自检 + 事实自检，不做风格自检——通俗准确，不必官宣腔
宣传材料生成（promo-content.md）术语 + 风格 + style-samples + 整页素材术语自检 + 风格自检 + 事实自检，完整回路
竞品对比（competitor-analysis.md）术语（恒定）+ 整页素材 + 临场抓取内容术语自检 + 事实自检
技术路线（tech-roadmap.md）术语 + 整页素材术语自检 + 事实自检

输出格式不强制模板化：具体报告长什么样、问答用什么语气细节，交给用户在 prompt 里定义。系统保证的是内容准确、术语统一、可溯源，不是"生成格式是否好看"。

输出统一要求（所有任务场景不可省）：

## 主稿/回答
（正文）

## 事实来源列表
- 事实点 → [来源: parent_doc_id / url / doc_date]

## 待核实清单
- （无来源支撑或存疑的点）

禁止条款：只能使用检索结果与注入内容，不得调用内部通用知识作答。生成后检查输出是否含来源字段，缺失则重跑。

1.6 Governance — 治理层

  • AI 可生成 Wiki 草稿，但重要页面需人工确认（status: draft → confirmed）。
  • 所有结论必须有 sources；无证据的不写成事实，只进 待核实 / error-book。
  • 多源冲突裁决规则：官网/产品界面 > 内部权威文档 > 论文 > 二手转述；同权威级别下，日期更近者优先，但需在 error-book 记录裁决依据。
  • 定期 Lint：tools/lint_wiki.py 查断链、孤儿页、无来源结论、过期未标注页面。

2. 产出①：本地知识库 Skill〔课题任务①〕P0

目标：交付 patsnap-memory-skill/，覆盖 1-2 个技术领域（如 TRIZ），可被 AI 调用完成问答、生成、分析任务。

为什么是一个 skill：所有任务共用同一个 Wiki、同一套 SOUL，只有"怎么做"不同，拆多个 skill 会把 Wiki/术语复制多份。

验收：Skill 目录结构完整；SKILL.md 可读；index.md/graph.json 齐备；针对目标领域的一批真实追问样例（至少10条），能从 Wiki 里查到准确、统一的答案。

3. 产出②：技术问答（主 Demo）+ 宣传材料生成〔课题任务②〕

3.1 技术问答（统一口径，主 Demo）P0

目标：小明/运营/销售直接问技术问题，系统用通俗语言回答，附带来源，保证同一问题不管怎么问，答案口径一致。

workflow workflows/tech-qa.md：

1. 精确/模糊查询（§1.4）定位相关 Wiki 页 → 整页加载
2. 术语自检（§1.3 术语节，恒定）
3. 事实自检：回查 source_url + doc_date
4. 输出通俗回答 + 来源列表（不做风格自检）

验收：同一问题用 3 种不同问法问，答案的核心结论一致（这是"口径统一"最直接的验收方式）；带来源；术语准确。

3.2 宣传材料生成P1

目标：生成公众号文章、产品介绍、海报文案、销售话术、活动推文、客户案例初稿等，贴合智慧芽官宣表达。

workflow workflows/promo-content.md：

1. 按用户 prompt 指定的素材范围，加载相关 Wiki 页整页
2. SOUL 完整注入（术语+风格）+ style-samples
3. 生成初稿 → 术语自检 → 风格自检 → 事实自检 → 改写
4. 输出（格式由用户 prompt 决定）+ 来源 + 待核实

验收：给定一个真实素材范围，能生成至少 2 种不同格式的材料（如公众号文章 + 销售话术），口吻贴合官宣风格；带来源。

4. 产出③：竞品分析与技术路线判断〔课题任务③④〕

4.1 竞品与技术对比（含临场 URL 抓取）P0

目标：销售甩一个竞品网址，系统当场抓取、对比、给出优势切入点——这是本产品差异化最强的能力，投入优先级和主 Demo 同级。

workflow workflows/competitor-analysis.md：

1. 用户提供竞品 URL（临场）或已有竞品名（走本地 wiki/competitors/）
2. 临场 URL：tools/fetch_competitor.py 实时抓取解析（§1.4）；失败则明确告知原因
3. 本地取证：混合检索智慧芽 product-capability 相关页 → 整页加载
4. 按维度对比生成结论 + 优势切入点
5. 高价值结论可选择性写回 wiki/competitors/（用户确认）
6. 自检 + 溯源

输出：对比结论（维度 | 智慧芽表现 | 竞品表现 | 优势-不足 | 证据来源）+ 宣传切入点建议。

验收：现场输入一个真实竞品 URL，能在合理时间内（分钟级）产出有理有据的对比；抓取失败时有清晰错误提示，不编造。

4.2 AI 技术突破路线判断P2

workflow workflows/tech-roadmap.md：

1. 本地盘点目标领域现有能力（整页加载）
2. 网络取 AI 趋势要点
3. 找结合点，输出候选路线，写入 roadmaps/ 页

输出：候选路线，每条含业务价值/技术基础/落地难度/优先级。

5. 评估：三方对比 P1

三方：①通用大模型（无外部知识）②普通 RAG（碎片检索，无术语恒定注入，无口径统一机制）③Wiki-Compiled RAG（完整方案）。

核心指标（呼应 §0.3 痛点优先级）：

  1. 口径一致性：同一问题 3 种问法，答案核心结论是否一致（这是本方案最该赢的指标，也是普通 RAG 最容易输的指标——它每次召回的碎片可能不同）。
  2. 事实准确性、术语一致性（用 eval_judge.py 打分）。
  3. 风格贴合度（仅评估宣传材料生成任务）。
  4. 改动量（反向，人工核算改动句子数/总句子数比例）。

验收：产出真实分数对比表；能具体说明"口径一致性"这项普通 RAG 和本方案的差距原因。

8.2 已知风险与应对

风险应对
目标竞品网站反爬/结构复杂，临场抓取不稳备 1-2 个已验证稳定的竞品作演示首选，抓取失败时清晰提示而非静默假装成功
知识库范围窄，被质疑"能力单薄"用 §0.4 范围原则 + 商业计划书 §6.2 落地计划的"可扩展叙事"回应：故意窄而深，不是能力不够
资料更新导致新旧结论冲突§1.2 时间线管理 + §1.6 权威性裁决规则 + superseded_by 链条
风格自检误套进技术问答场景，答案显得"端着"§1.5 按任务场景严格区分加载内容，问答场景不加载风格节
现场演示技术问答被追问超出知识库覆盖范围提前用 §2 验收里的"真实追问样例"批量测试，找出知识盲区，优先补齐或提前准备"待核实"式坦诚回应

8.3 理论依据

  • Wiki 是解决"口径统一"的机制，非单纯"检索优化"：普通 RAG 每次现找现拼、两次问法可能召回不同碎片，答案不稳定；LLM Wiki 编译出唯一标准答案页，检索命中的是"同一页"，口径天然统一。
  • 整页加载：检索定位页面，命中后加载整页而非碎片，保证问答/生成时信息不漏段。
  • 不引入 SAG：本产品无因果链多跳追根因需求，轻量 graph.json 够用。
