一、第一性原理：这道题要解决什么

先定义问题，再选技术

题目三个痛点是风格漂移、术语失准、经验断层，对应三种能力：企业记忆、企业表达、企业判断。翻译成技术语言得到四条约束，它们决定后面每一个技术选择。

约束1 · 指定素材任务，非开放问答

宣传稿针对选定项目，竞品对比针对指定竞品，路线判断针对给定能力盘点。素材是"点名"来的。所以取材以元数据确定性圈定为主，语义检索为辅。

约束2 · 个性化发生在生成，不在检索

让输出"像智慧芽"的是术语表、风格指南、工作流，它们是恒定注入的上下文。检索召回再准也不会让文案变成官宣口吻。

约束3 · 双数据源是硬要求

任务②③④明写"结合网络数据"：竞品资料不在本地、趋势判断要看当下。系统必须有本地库 + 网络检索两条取材通道。

约束4 · 可溯源是可信度前提

竞品对比、路线判断要经得起追问。每个事实点绑回来源，无来源的标"待核实"，而不是让模型编。

由此得到贯穿全文的取舍原则：检索层是"普通 RAG"对照组也有的能力，堆重它不产生差异化。工程力应投在检索之上——知识组织、个性化注入、自检改写、溯源。

二、整体架构

六段链路

系统是一条六段管线。与通用 RAG 的差别集中在第 2、4、5 段：知识怎么组织、生成前注入什么、生成后如何自检。

① 数据接入→ ② 知识组织→ ③ 检索取材（本地＋网络）→ ④ 个性化注入→ ⑤ 生成＋自检→ ⑥ 溯源＋评估
各段对应章节：① → 三；② → 四（三条路线在此分野）；③ → 五；④ → 六；⑤⑥ → 七、八。第九节讲工程部署与路线选型。

三、数据接入

解析 → 代码处理 → 切片 → 元数据

数据接入产出两种形态：供事实检索的切片，和供风格学习的整篇文档。同一份资料按用途可同时产出两者。

1

解析 · Smart-Doc API

三个端点：/image/smart-doc（图片）、/pdf/smart-doc（PDF）、/document/smart-doc（Office 文档，先转 PDF 再处理，支持 doc/docx/ppt/pptx/xls/xlsx 等）。调用参数置 detailed:true, markdown:true, ocr_backend:"auto"。

先讲清它返回什么、不返回什么：它做的是版面分析 + OCR + Markdown 聚合，产出拼接好的 Markdown 文本和带 bounding box 的版面块。它不直接给"H1/H2/H3 树"或"表格行列关系"的结构化 JSON——标题层级和表格要从它输出的 Markdown 里自己解析（Markdown 的 #/## 和表格语法已够还原）。这一步只写一个适配器：Markdown → 统一内容块。

2

代码文件 · 单独处理（Smart-Doc 不管代码）

开源项目代码用 tree-sitter 解析，以"一个函数 + 签名 + docstring"为一块，函数体细节丢弃。不要把整个文件揉成一块，否则问具体能力时定位不准。README/文档仍走 Smart-Doc。

3

切片 · 按语义单元，且区分用途

事实检索用的切片：按标题层级和语义段落切，每块几百字，一块一个完整意思。

关键：风格样本不切片。风格藏在整篇文章的谋篇、节奏、过渡里，切碎就没了，而风格是本课题主轴。官宣/技术博客整篇另存一份供第六节风格注入；同一批文案再切一份进检索库供事实使用。

4

元数据 · 本课题的检索主键

每块强制打标——"指定素材"任务靠这些字段确定性圈定范围：

{ "chunk_id": "eureka-readme-0007", "parent_doc_id": "eureka-readme", "source_type": "official | paper | code | doc | competitor", "project": "eureka", "usage": ["技术传播"], "title": "Eureka 产品说明", "year": 2024, "source_url": "https://...", "content": "……", "embedding": [/* 1024 维，路线 A/C 用 */] }
力气分配：真正决定质量的是切片和元数据，靠人理解、不能自动化偷懒。解析用官方 API、不自研 OCR。评委一眼能看出知识库是"精切精标"还是"整本塞进去"。

四、知识组织：三条技术路线

方案的分野点

数据接入之后，"知识怎么组织"有三条路线。它们不是优劣关系，而是投入产出与叙事高度的权衡。逐条给出做法、代价、适用，最后给推荐。

路线 A

纯向量 RAG

最稳 · 最省事

做法：切片 → embedding → 向量库；查询时向量 kNN + BM25 混合召回，取 topK 塞进上下文。经典 RAG。

代价：低，一周内一定做得完，工具链成熟。

问题：这正是三方对比里"普通 RAG"对照组的样子。只做 A 等于把自己做成对照组，差异化全靠第六节注入层撑，叙事也平。

适用：作为保底基线和对照组必然要有；但不宜作为最终形态和主叙事。

路线 B

LLM Wiki 为主，向量为辅

推荐 · 立意最高

做法：把资料编译成互链的 Wiki 页面——项目一页、论文一页、技术能力一页、术语一页、竞品一页、Demo 场景一页，页面间双向链接。Agent 操作从"retrieve chunks"升级为 search / read / follow-link：先搜到入口页，再顺链接遍历相关实体。向量检索退为"帮 Agent 找到该读哪一页"的入口定位器。另设 Error Book 页沉淀错误与待核实项。

为什么更贴题：直接兑现三种能力——页面结构=企业记忆，风格页/术语页=企业表达，竞品页/路线页=企业判断。"受控生长"（AI 辅助建页、人工审核、修正写回）落地详解里"越用越懂"的复利叙事。主叙事可打成 "从 RAG 到 LLM Wiki：构建智慧芽企业记忆"。

代价：中。要设计页面模板、建页、维护链接，但页面可由 AI 辅助生成 + 人工确认，一周做 15-25 页样板可行。

企业版必须受控：不能让 AI 随便写改。重要页面人工确认、核心分类人工审核、不确定项只标注不写成事实、修正保留来源与版本。定位为受控的场景型企业 Wiki（技术传播与战略分析），不是"全公司大百科"——后者治理与合规风险会被质疑。

路线 C

双层：Wiki 在上，向量在下

最完整 · 最重

做法：知识层用 Wiki 页面组织（人读、Agent 遍历、可溯源）；检索层用向量库做页面内与跨页的语义定位。Wiki 负责"知识的结构和记忆"，向量负责"在结构里快速找相关片段"，各司其职。

代价：高。等于 B 和 A 都搭还要打通，一周半脱产下工作量最大。

适用：数据规模大、或想在答辩同时展示"结构化记忆 + 工程完整度"时。时间紧则不必强求，B 已能讲清立意。

选型判断：A 一定要有（保底 + 对照组）。主推 B：立意最高、最贴题、工作量可控——Wiki 组织知识，向量库降为入口定位器，你们已投入的向量库不浪费、且被放到更合理的位置。C 作为有余力的目标。落地路径：先按 A 跑通保底，再把知识层重构成 B 的 Wiki 结构——A 的向量库正好被 B 复用为入口定位器，两步是叠加而非重做。
知识层目录（路线 B/C）： patsnap-memory-skill/ ├── SKILL.md ├── wiki/ │ ├── projects/ # 每个开源项目一页 │ ├── papers/ # 每篇论文一页 │ ├── technologies/ # 每个核心技术能力一页 │ ├── competitors/ # 每个竞品一页 │ └── use-cases/ # 每个 Demo 场景一页 ├── glossary.md # 术语页（表达层） ├── style-guide.md # 官宣风格页（表达层） ├── workflows/ # 三个任务各一套分步流程 ├── error-book.md # 错误与待核实项沉淀 └── evaluation-card.md # 评估卡
出处表述留分寸：LLM Wiki 概念受 Karpathy 倡导启发，但原帖全文未直接核验。正式材料建议表述为"受 Karpathy 所倡导的 LLM Wiki 思路启发，结合近期 agent-native retrieval / self-evolving wiki 研究"，不要把出处说得过满。

五、检索：元数据圈定为主，语义为辅

三个任务形状不同，一个开关切换

因为是"指定素材"任务，检索第一步不是撒向量，而是用元数据把范围锁死，再在范围内做语义排序。三个任务的检索形状差异很大：

宣传稿（主）· 检索最轻

点名一个项目，其全部资料通常小到能整篇注入上下文。最优解是全量注入、几乎不检索——检索反而可能漏段。

竞品对比 · 检索最重

按多维度取证，竞品资料主要在网络。这里本地库 + 网络双源才真正值回票价，是向量/Wiki 检索发挥最大价值处。

技术路线 · 网络为主

依赖当下 AI 趋势，本地只提供"现有能力"这一半，另一半靠联网与人的判断。检索是辅助。

一个开关解决：管线支持"全量注入"与"检索取材"两档。主 Demo 走全量注入档，竞品/路线走检索+网络档。范围锁定后，范围内再走向量 kNN + BM25 融合（A/C）或 Wiki follow-link（B）。收益主要来自"范围圈得对"，不是"权重调得巧"，不要在权重上过度调参。
网络检索的边界：竞品/路线两 Tab 按主方案是"提前跑好、人工校对"的结果，故网络检索是准备阶段工具，不做现场实时联网（慢且不可控）。网络结果同样带 source_url 落进溯源，与本地一视同仁；冲突时以本地权威为准并标注。联网只取公开信息，不外传私域内容。

六、个性化注入：个性化真正发生处

术语 / 风格 / 工作流，恒定注入

前面拿到的还只是"料"。让输出"像智慧芽"的是生成前强制叠加的三层恒定上下文——不随查询变化、每次都在，是"专属 Skill"区别于"普通 RAG"的地方。

术语层

10-20 条"标准叫法 / 常见误用 / 推荐译法"对照表，全量注入，生成前先校准命名。体量小、不检索。条目须拿智慧芽真实用法逐条核实，不能凭印象写。

风格层

5-10 条可执行风格规则 + 2-3 篇整篇真实官方文案作"改写标尺"，让生成对齐风格而非自由发挥。整篇样本来自第三节不切片的那份。

工作流层

该任务的分步流程作为指令骨架：检索→提取→规划→草稿→术语校准→风格改写→事实校验→输出，让模型按流程办事、每步可追溯。

最终 Prompt 组装顺序：工作流指令（怎么做）＋ 术语表＋风格规则与样本（按谁的标准）＋ 检索到的事实素材（用什么料）＋ 用户任务。检索结果只是一块料，三层注入才是"厨师的手艺"。

七、生成与自检：把"能跑"推到"效果好"

批判回路 + 溯源

一次性生成的输出，风格会漂、术语会错。真正拉高质量的是生成后让模型拿着术语表和风格样本自我批改、再改写一遍。这条回路的收益大于任何检索调优。

生成初稿→ 术语自检（对照术语表逐条查）→ 风格自检（对照样本打分）→ 事实自检（每点绑来源）→ 改写终稿

溯源

输出结构固定为主稿 + 事实来源列表 + 不确定项清单。每个关键事实点绑回 source_url / parent_doc_id；无来源支撑的明确标"待核实"。路线 B 的 Error Book 收集这些待核实项，人工确认后写回 Wiki。

禁止条款

Prompt 写死"只能使用提供的检索结果与注入内容，不得调用内部知识"；每次调用检查返回是否含来源字段，防止模型跳过标注。这是可信度的工程保障。

为什么这才是效果的来源：检索决定"有没有料"，这条回路决定"料写得像不像、准不准"——后者才是评委打分时真正在看的东西。

八、评估：三方对比

把"感觉更好"变成"数字证明"

同一任务用三种方式各跑一遍：通用大模型 / 普通 RAG（路线 A）/ 专属方案（B 或 C + 三层注入），按四项指标打分。加"普通 RAG"这一档，直接堵住"你们不就是个 RAG"的质疑。

指标定义打分方法
事实准确性陈述是否有资料支撑、有无编造。标出关键事实点逐条对照，算命中率。
术语一致性是否符合智慧芽标准命名。用术语对照表统计命中率。
风格贴合度语气句式是否像官方文案。与真实官方文案并列，用另一模型或 2-3 人按 1-5 分评。
改动量（反向）改到可直接发布还需人工改多少。熟悉风格的人当编辑，记录改动字数占比，越低越好。
对照组几乎零成本："普通 RAG"就是"本方案砍掉三层注入和自检回路后的样子"。所以对照组直接复用同一套管线、关掉注入即可——对比最干净，且证明"多出的分正是注入与自检带来的"。用真实跑出的数据，样本 5-8 组即可，不要用漂亮的假数。

九、工程与选型

部署、路线取舍、风险

部署

组件

  • 解析：官方 Smart-Doc API（内网直连，结果本地缓存，同文档不重复解析）。
  • 向量库：任一支持"元数据过滤 + 向量检索"的库；万级切片以内单节点足够，不上集群。
  • 向量化：离线批量跑，不常驻 GPU。
  • 生成：主流大模型 API。

与产品的关系

  • 本方案是详解里"网页产品"的引擎：主 Demo（宣传稿）现场真跑通这条链路。
  • 竞品/路线两 Tab 展示准备阶段跑好、人工校对的结果。
  • 知识层（路线 B 的 Wiki 目录）就是可交付的 Skill 文件夹。

路线取舍一览

路线工作量贴题/立意用途
A 纯向量 RAG低低（就是对照组）保底基线 + 三方对比对照组，必做
B Wiki 为主中高（企业记忆 + 复利叙事）方案主形态与主叙事，推荐
C 双层高高 + 工程完整度数据量大/秀工程时的进阶目标

主要风险与应对

风险

  • 私域数据大部分是公开的，"企业记忆"最弱。
  • Wiki 建页耗时，可能挤占主 Demo。
  • 现场实时链路崩。

应对

  • 主轴押"企业表达/术语"（公开数据可支撑），记忆尽力而为。
  • Wiki 先做 15-25 页样板，AI 辅助生成 + 人工确认。
  • 主 Demo 备"预置结果"降级档；先按 A 跑通保底再升 B。
一句话选型：A 保底、B 主推、C 选做。向量库你们保留得对——它在路线 B 里降为"入口定位器"、在竞品场景真有用，只是别把它当胜负手。真正拉开差距的是知识组织成 Wiki、三层注入、生成后的自检回路。
