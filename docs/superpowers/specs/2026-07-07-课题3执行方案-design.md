# 课题3 执行方案 · 设计规格（Spec）

> **本文用途**：这是一份施工图纸，读者是未来的 Claude 会话——它没有当前对话的记忆，只能依据本文把项目做出来。因此凡"怎么做"的技术决策都写到可执行粒度；凡认知/判断类工作，写成给 Claude 的策展指令，而非黑盒 API 调用。
>
> **配套文档**：本 spec 定稿后由 writing-plans 产出对应的 implementation plan（工序与验证）。本文只回答 what & why；执行顺序见 plan。
>
> **展示向文档**：`技术方案.html`（技术叙事）、`课题详解.html`（全局作战手册）、`LLM-Wiki-概念笔记.md`（路线B概念依据）。本 spec 是它们的施工版，冲突时以本 spec 为准。

## 0. 项目上下文与约束

### 0.1 课题（硬边界，不可偏离）

课题：**基于本地私域数据的 AI 个性化能力构建**。三个痛点：风格漂移、术语失准、经验断层。翻译为三种能力：**企业记忆、企业表达、企业判断**。

**三个期望产出（本 spec 的主干，每一章对应一个）：**

1. **本地知识库 Skill** — 基于智慧芽本地资料，可被 AI 调用完成内容生成和分析任务。（课题任务1）
2. **开源项目宣传材料** — 结合本地知识库与网络数据生成的富文本，贴合官宣表达。（课题任务2）
3. **技术对比与路线判断** — 竞品技术优劣分析 + 智慧芽 AI 技术突破路线判断。（课题任务3+4）

**可用资源**：主流大模型 API + Vibe Coding 工具；Smart-Doc 文档解析 MCP；智慧芽公开论文与开源项目。

### 0.2 核心架构：Wiki-Compiled RAG

一句话定调：**不是 Wiki 替代 RAG，也不是 RAG 加个 Wiki 壳，而是"编译后的 RAG"——先把散落的原始资料离线编译成结构化的企业记忆 Wiki，运行时再用混合检索召回 Wiki 页面（整页，非碎片），最后经 Skill 工作流生成业务产出。**

这从第一性原理推出，五个基本事实决定了它：

1. 模型不知道企业私域知识 → 必须外部注入。
2. 上下文窗口有限 → 不能全塞，必须检索。
3. 原始资料散、旧、冲突、重复 → 直接 RAG 会召回脏知识，检索前要先"编译"。
4. 企业任务要的不只是事实，还有风格和判断 → 光"查资料"不够。
5. 企业答案必须可追溯 → 不能写得漂亮却不知依据在哪。

由此得到五层架构（本 spec §1 逐层展开）：

```
┌─ 1.1 Raw Sources    原始证据层（只读、不可变，事实底座）
├─ 1.2 LLM Wiki       企业理解层（编译后的结构化页面 + index + graph）
├─ 1.3 SOUL           企业人格层（术语 + 风格，恒定注入）
├─ 1.4 Retrieval      混合检索层（意图路由 → 整页加载）
├─ 1.5 Generation     生成与自检（工作流 + 溯源）
└─ 1.6 Governance     治理层（受控生长、写回、Lint）
```

**关键分工**（贯穿全文）：
- **RAG 解决"运行时去哪找"**，**Wiki 解决"平时怎么存"**。
- **Raw Sources 永远是事实底座，Wiki 只是编译层**——每个结论都能从 Wiki 页回查到原始资料。
- **确定性工作交给脚本**（解析、建索引、扫描建图、校验），**认知工作交给 Claude**（编译页面、策展、生成、判断）。

### 0.3 关键约束（写死，防跑偏）

- **知识组织 = LLM Wiki**：资料编译成互链 Markdown 页面，agent 通过 index → 读页 → follow-link 使用。
- **检索 = Wiki-Compiled RAG**：向量/关键词只用于"定位到哪一页"，命中后**加载整页**（见 §1.4 为什么整页）。
- **不引入 SAG / 重型知识图谱**：课题三个任务没有"因果链多跳追根因"的需求（宣传稿要全貌、竞品对比填矩阵、路线做盘点），只需轻量 `graph.json` 做够用的关系扩展。理由见 §1.2。
- **数据以公开为主**：论文/开源/官宣/竞品公开资料。企业记忆能力尽力而为，不硬吹。
- **主轴押"企业表达 + 术语"**：公开数据最能支撑风格与术语，主 Demo 是宣传稿（风格驱动）。
- **可溯源**：每个事实点绑回来源；无来源支撑标"待核实"，不编造。
- **竞品对比内容做真**：真查本地能力 + 真查网络竞品 + 真产出带证据矩阵；现场展示预置的真实结果，不做现场实时联网。

### 0.4 开工前必须先确认的输入（缺则先问，不许瞎猜）

Claude 动手前先核对下列输入是否齐备。任一缺失，先向用户索取，不要用占位数据继续：

- [ ] **语料清单**：论文/开源项目/官宣文案的实际文件或 URL（公开为主）
- [ ] **主 Demo 目标项目**：宣传稿针对哪一个开源项目（点名一个）
- [ ] **竞品**：对比针对哪 1-2 个竞品
- [ ] **Smart-Doc**：`BASE_URL` 与调用凭证/鉴权方式
- [ ] **大模型 API**：生成用的 key 与模型名
- [ ] **向量库**：选型（哪个库、本地起还是云）、连接信息
- [ ] **术语真值来源**：可核对智慧芽标准术语的官网/产品界面/论文入口（术语表严禁凭印象编）

## 1. 五层引擎（三个产出共享的底层）

### 1.1 Raw Sources — 原始证据层

**定位**：只读、不可变的事实底座。Claude 只从这里读，绝不改写。Wiki 层的每个结论都要能回查到这里的某份原始资料。

**只有确定性环节写成脚本**，理解/切分/判噪由 Claude 做。

**`tools/parse_docs.py`** — Smart-Doc 适配器
- 三端点：`/image/smart-doc`、`/pdf/smart-doc`、`/document/smart-doc`（Office 先转 PDF）。
- 请求参数：`detailed:true, markdown:true, ocr_backend:"auto", sign_url:false, reject_aspect_ratio:false`；PDF 加 `pdf_render_dpi:200, pdf_page_parallelism:4`。
- 响应是流式字符串块，`markdown:true` 时最后一块是拼接好的 Markdown → 落地到 `raw/<doc_id>.md`。
- **必须实现本地缓存**：以文件内容 hash 为 key，同文档不重复调用 API。
- **重要事实**：Smart-Doc 返回 Markdown + bounding box，**不返回结构化的 H1/H2/H3 树或表格行列 JSON**。标题层级和表格由后续从 Markdown 语法自行解析。适配器只负责"拿到干净 Markdown + 缓存"。

**`tools/parse_code.py`** — 代码提取
- tree-sitter 解析开源项目代码；一个块 = 一个函数（签名 + 参数 + docstring），函数体丢弃。
- README/文档不走这里，走 parse_docs。

**Claude 做的（非脚本）**：读 `raw/` 的 Markdown → 判断并去除噪声（页眉页脚/导航/乱码/调试残留）→ 按标题层级和语义段落切分（几百字/块，一块一个完整意思）→ 判定风格样本（官宣/技术博客整篇保留，进 style-samples/）→ 为每块补元数据。

**切片元数据结构**（供 build_index 和溯源用）：
```json
{
  "chunk_id": "eureka-readme-0007",
  "parent_doc_id": "eureka-readme",
  "source_type": "official|paper|code|doc|competitor|trend",
  "project": "eureka",
  "usage": ["技术传播"],
  "title": "Eureka 产品说明",
  "year": 2024,
  "source_url": "https://...",
  "content": "……"
}
```

**验收**：给定一份 PDF/README，跑 `parse_docs.py` 得到 `raw/` 下干净 Markdown（二次调用命中缓存）；Claude 能据此产出带完整元数据的切片，风格样本另有整篇副本。

### 1.2 LLM Wiki — 企业理解层

**定位**：把 Raw Sources 编译成结构化、互链、可溯源的 Markdown 页面。这是 Skill 的知识主体。**它是编译层，不是事实源**——事实源永远是 raw/。

**为什么是"编译"而不是"堆文档"**：直接 RAG 每次从原始碎片现找现拼，不改变知识本身的散/旧/冲突状态。LLM Wiki 在检索前加一道编译：把散落矛盾的材料预加工成 AI 可直接消费的语义单元。这是本方案区别于普通 RAG 的根本。

**页面类型**（`wiki/` 下）：
- `projects/` 每个开源项目一页
- `papers/` 每篇论文一页
- `technologies/` 每个核心技术能力一页
- `competitors/` 每个竞品一页
- `roadmaps/` AI 技术路线专题页
- `use-cases/` 每个 Demo 场景一页

**页面模板**（frontmatter 承载结构化关系字段，正文承载语义）：
```markdown
---
title: Eureka
type: project            # project|paper|technology|competitor|roadmap|use-case
status: draft            # draft|confirmed（受控生长：AI 建初稿，人工确认后改 confirmed）
confidence: high         # high|medium|low
sources:                 # 溯源：本页每个事实回指 raw/ 的哪份材料
  - {ref: eureka-readme, url: "https://..."}
related:                 # 结构化关系（脚本据此建 graph.json，不靠解析正文）
  - technologies/知识图谱
  - papers/xxx-2024
audience: [developer, marketing]
updated: 2026-07-07
---

## 概述
（正文，每个事实点后标 [来源: parent_doc_id]）

## 相关
- [[technologies/知识图谱]]   ← wikilink，正文里的弱关联，graph 兜底扫描

## 待核实
- （不确定的内容放这里，不写进正文当事实）
```

**页面必须"短而聚焦"**（这条决定整页加载能否成立，见 §1.4）：
- 一页只讲一个实体，别把多个项目塞一页。
- 正文是"编译过的理解"，不是把原始文档整篇复制进来。
- 单页超过约三四千字就拆页，用 `[[链接]]` 连。

**两个特殊导航文件**（LLM Wiki 检索的关键，别遗漏）：
- **`index.md`** — 全局目录，面向内容。列出每页 + 一句话摘要 + 分类。检索第一步先读它定位相关页。这在中等规模（数百页内）下效果很好，无需依赖向量库。
- **`graph.json`** — 轻量关系图，由 `build_graph.py` 扫描所有页 frontmatter 的 `related` 字段 + 正文 `[[wikilink]]` 生成。节点=页面，边=关系。供"相关技术扩展"这类多路召回用。

**为什么用 graph.json 而非 SAG/知识图谱**：SAG 那套（event/entity + SQL 动态多跳）是为"顺着因果链追根因"设计的（如"指标为什么降→追到供应商"）。课题三任务没有这种需求——宣传稿要单实体全貌、竞品对比横向填矩阵、路线做能力盘点。最多需要"从一个技术找到相关几个技术"的轻度扩展，graph.json 沿边走一两跳就够。不为用不到的能力上重型设施。

**策展指令（Claude 执行）**：
1. 读 §1.1 产出的切片与整篇文档。
2. 每个实体建一页，用模板填写；`related` 字段写清结构化关系。
3. 事实点标 `[来源]`；不确定的进 `待核实` 段和 error-book，绝不写成事实。
4. 建 `[[双向链接]]`：项目→用到的技术→相关论文→对标竞品。
5. 新建页 `status: draft`，人工确认后改 `confirmed`。
6. 页面落盘后跑 `build_index.py`（切片入向量库）和 `build_graph.py`（建关系图）。

**验收**：wiki 每类至少有真实页面（首版 15-25 页）；`index.md` 覆盖全部页面；`graph.json` 能生成且边指向合法页面；每个事实有来源、可回查 raw/；不确定项在 error-book 而非正文。

### 1.3 SOUL — 企业人格层

术语层与风格层合并进一个 **`SOUL.md`**（企业的"语言人格"），运行时作为**恒定上下文**拼进每次生成（每次都在，不检索）。合一个文件更简明、叙事更强（"注入企业之魂"）。但**术语在 SOUL.md 里保持独立一节**，好让"逐条核实"有清晰抓手——术语是客观对错（错了在专家评委前翻车），风格是主观贴合，两者性质不同，不能糊在一起。整篇范文因体量大，单独放 `style-samples/`。

**`SOUL.md`（术语规范 + 风格规则）**

怎么做：
- 术语节——读语料时记下术语 → 逐条对照 §0.4 的术语真值来源核实 → 填表。**核实不了的不填，进 error-book**（严禁凭印象编）。
- 风格节——读 3-5 篇真实官宣 → 总结共同写法，写成**能照着改的动作**（不是抽象形容）。

```markdown
---
type: soul
updated: 2026-07-07
---

# SOUL — 智慧芽的语言人格

## 一、术语规范（客观，逐条核实过）
| 标准叫法 | 常见误用 | 推荐英文 | 来源 |
|---------|---------|---------|------|
| 专利同族 | 专利家族 | patent family | 官网/产品界面（待核实） |
| 智慧芽   | 智慧牙   | PatSnap        | 官网 |
| …（10-20 条，每条都核实过，来源列写清出处） |

## 二、表达风格（主观，对照样本改写）
1. 句式偏短，一句一个意思，避免从句套从句
2. 避免营销腔：不用"业界领先""颠覆性"这类形容词
3. 技术描述带具体数据：不用"大幅提升"，要用"提升 40%"
4. 主语习惯用"帮助企业/团队…"
5. …（每条都是可落地的改写动作）
   整篇范文见 style-samples/（体量大，此处只放规则）
```

**`style-samples/`（整篇官宣范文，风格标尺）**：把 2-3 篇**整篇**真实官宣原文存入，一篇一个文件。**整篇不切片**——风格藏在谋篇、过渡、节奏里，切碎就没了。写稿时 AI 整篇读它，模仿句式与节奏。

### 1.4 Retrieval — 混合检索层

**核心思想：检索只负责定位"读哪几页"，不负责产出答案素材。命中后加载整页 Wiki（非碎片）。** 这就是 Wiki-Compiled RAG，也是它和普通 RAG 的根本区别。

**什么是"整页加载"**：
- 普通 RAG：检索直接返回最相似的几个 chunk（碎片），模型只看到被切下的段落，容易漏关键段、也不知道段落间关系。
- 整页加载：检索（index/BM25/向量）只用来**定位到"是哪一页"**，命中后加载**整个 Wiki 页面**给模型。检索结果是"路标"不是"素材"。
- 为什么不怕冗余：Wiki 页面是**编译过的精炼语义单元**（一两千字/页），不是原始长文档；加上"先读 index 只取相关页、一次 3-6 页、分层呈现"的控制，冗余可控，换来的是"不漏关键段 + 上下文完整"。这个交易对"写宣传稿、做判断"这类需要全貌的任务划算。（前提：§1.2 要求页面短而聚焦。）

**检索按意图路由**（不要用一种方式套所有任务）：

```
意图识别
├─ 精确查询：用户点名实体（项目名/论文名）
│    → 读 index.md 定位 → 直接加载该页整页 + related 页
│    → 不走向量库（已知道要哪页，直接取更准、不漏段）  ← 主 Demo 宣传稿走这条
│
├─ 关系查询：实体名 + 方向/关联词（相关技术、对标竞品）
│    → 沿 graph.json 边遍历（一两跳）→ 加载相关页整页  ← 竞品对比的"相关技术扩展"
│
└─ 模糊查询：关键词/概念/问句（"智慧芽做过图数据库吗"）
     → 先读 index.md 把范围收窄到 2-3 类
     → 混合检索：BM25（精确术语）+ 向量（语义相近），RRF 融合，粗筛出候选页
     → 加载候选页整页 → 分层呈现（先结论后证据）  ← 语义问答、竞品本地取证走这条
```

**向量库的定位（回应"数据量"）**：
- 本次比赛语料不大时，`index.md` + grep already 够用；但**向量库照建**——(1) 数据会随企业场景增长，向量库是规模化的必需件；(2) 三方评估里"普通 RAG"对照组本来就需要一个向量检索系统；(3) 模糊查询这条线本就依赖它。
- `build_index.py` 把切片**逐块**灌入向量库（每条记录 = 一个切片）。但注意：**向量库返回的是"命中哪些页"的信号，最终仍加载整页**，不是把碎片直接喂模型。

**网络检索（线：Web）** — 竞品对比、技术路线（课题②③④明写"结合网络数据"，硬要求）：
- `tools/web_search.py` 并联联网，取竞品/趋势公开信息，带 `source_url` 落溯源。
- 与本地冲突时以本地权威为准并标注；只取公开信息，不外传私域内容。
- **不做现场实时联网**：竞品/路线是准备阶段跑好、人工校对的结果。

### 1.5 Generation — 生成与自检回路

写成 Claude 的执行流程，不是黑盒 API。生成后必须自我批改再输出。

**运行时 Prompt 组装顺序（固定）**：
```
[工作流指令]  怎么做（本任务的分步流程，来自 workflows/）
+ [SOUL]      术语按标准命名 + 按谁的口吻写（恒定注入）
+ [整页素材]  §1.4 加载的完整 Wiki 页（每条事实带 source_url）
+ [用户任务]  具体要生成什么
```
检索到的整页是"料"，SOUL 是"手艺"。

**自检回路**：
```
生成初稿
→ 术语自检：逐条对照 SOUL.md 术语节，改正误用命名
→ 风格自检：对照 SOUL.md 风格节与 style-samples/，改写不符处
→ 事实自检：每个关键事实点绑回 source_url（回查 raw/）；无来源支撑的标"待核实"
→ 改写终稿
```

**输出固定结构**（所有生成任务统一）：
```
## 主稿
（正文）

## 事实来源列表
- 事实点 → [来源: parent_doc_id / url]

## 待核实清单
- （无来源支撑或存疑的点，进 error-book）
```

**禁止条款**（写进每个生成 prompt）："只能使用提供的检索结果与注入内容，不得调用你的内部通用知识作答"。生成后检查输出是否含来源字段，缺失则重跑。

### 1.6 Governance — 治理层

企业版必须有这层，否则会被质疑"不可信"。规则从简：
- AI 可生成 Wiki 草稿，但重要页面/核心分类需人工确认（`status: draft → confirmed`）。
- 所有结论必须有 `sources`；无证据的不写成事实，只进 `待核实` / error-book。
- 生成与判断分离：需要主观推断的字段（如页面归类）先留空，基础页面落盘后再独立判断，人工确认。
- 多源冲突时以更权威来源为准（如代码/官网 > 二手转述），并标注。
- 定期 Lint：`tools/lint_wiki.py` 查断链、孤儿页、无来源结论、术语不一致、过期页面；不通过项进 error-book 待治理。

**这层也是"可复制性 SOP"的载体**：确定性环节沉淀为 tools/ 脚本 + 页面模板 + 本 spec 的策展步骤。换企业只需换 corpus/ 语料、重跑同一套 SOP。对应商业计划书的"通用底座 + 专属配置"。

## 2. 产出①：本地知识库 Skill 〔课题任务1〕〔P0〕

**目标**：交付一个可被 AI 调用完成生成与分析任务的 Skill——即 `patsnap-memory-skill/`。

**切题锚点**：课题任务1"设计本地知识库 Skill"、期望产出1"可被 AI 调用"。

**为什么是一个 skill，不是三个**：三个任务（宣传稿/竞品/路线）共用**同一个 Wiki、同一套 SOUL**，只有"怎么做"不同。知识与人格全共享、仅工作流不同——这正是"一个 skill、多个 workflow"的场景，也贴课题"设计本地知识库 Skill"（单数）。拆三个会把 Wiki/术语/风格复制三份。叙事上也更强：不是三个工具，是**一个懂智慧芽的数字员工**，会写宣传稿、会做竞品分析、会判断技术路线。（跨企业规模化时才分"通用底座 skill + 专属配置"两层，那是商业计划书 §6.2 的故事。）

**实现规格**：
- 完成 §1.1 → §1.6（数据接入 → Wiki 编译 → SOUL → 检索 → 生成自检 → 治理）。
- `SKILL.md` 写清：这个 Skill 是什么、包含哪些页面/能力、如何调用（三个 workflow 入口）、如何溯源。
- 三个 workflow 就是被调用的能力入口。

**验收**：Skill 目录结构完整；SKILL.md 可读；任取一个 workflow，Claude 能据此产出带溯源的结果；index.md/graph.json 齐备。

## 5. 评估：三方对比 〔P1〕

**目标**：把"感觉更好"变成"数字证明"，堵住"你们不就是个 RAG"的质疑。

**切题锚点**：证明个性化差异化，支撑三个产出的说服力。

**三方**：
1. 通用大模型（无外部知识）
2. 普通 RAG（碎片检索 + 直接生成，无 SOUL、无自检、无整页加载）
3. Wiki-Compiled RAG + Skill（完整方案）

**四项指标**：事实准确性、术语一致性、风格贴合度、改动量（反向）。
**方法**：`tools/eval_judge.py` 用另一个大模型当裁判（除"改动量"需人工），5-8 组样本。

**关键设计**：对照组"普通 RAG"= 完整方案**关掉三样开关**（整页加载→退回碎片、SOUL 注入→关、自检→关）。同一套底层、只关开关，对比最干净，且证明"多出的分正是编译+注入+自检带来的"。这也让向量库有了明确用武之地（对照组的检索系统）。

**验收**：产出真实分数的对比表（非示意值）；能说清每项 5 分与 4 分差在哪。

| 前端泄露 API key | key 走后端代理 |

### 8.4 理论依据（ref/ 四篇）
- **Wiki-Compiled RAG（编译时 vs 运行时）**：`构建 AI 时代的知识底座：直播数据 LLM Wiki 实践` — RAG 每次现找现拼、不改变知识状态；LLM Wiki 在检索前加"编译"。
- **RAG 不会死、逐层互补**：`向量检索、知识图谱与 LLM Wiki` — 全文/向量/混合/图谱/Wiki 是补足非替代；LLM Wiki 到规模后仍需关键词/向量检索。
- **三种能力分工**：`SAG + LLM Wiki` — RAG 找事实、SAG/Graph 连关系、LLM Wiki 留理解。本方案取其"编译+可溯源+写回"骨架，但因课题无因果多跳需求，不引入 SAG。
- **整页加载工程折中**：`深度解析 LLM Wiki / GBrain` — 向量粗筛 → 整页加载 → 分层呈现；潜在空间（LLM 判断）vs 确定性（脚本执行）分工。
