# 课题3 执行方案 · 设计规格（Spec）

> **本文用途**：这是一份施工图纸，读者是未来的 Claude 会话——它没有当前对话的记忆，只能依据本文把项目做出来。因此凡"怎么做"的技术决策都写到可执行粒度；凡认知/判断类工作，写成给 Claude 的策展指令，而非黑盒 API 调用。
>
> **配套文档**：本 spec 定稿后由 writing-plans 产出对应的 implementation plan（工序与验证）。本文只回答 what & why；执行顺序见 plan。
>
> **展示向文档**：`技术方案.html`（技术叙事）、`课题详解.html`（全局作战手册）、`LLM-Wiki-概念笔记.md`（路线B概念依据）。本 spec 是它们的施工版，冲突时以本 spec 为准。

## 0. 项目上下文与约束

### 0.1 课题（硬边界，不可偏离）

课题：**基于本地私域数据的 AI 个性化能力构建**。三个痛点：风格漂移、术语失准、经验断层。翻译为三种能力：**企业记忆、企业表达、企业判断**。

**三个期望产出（本 spec 的主干，每一章对应一个）：**

1. **本地知识库 Skill** — 基于智慧芽本地资料，可被 AI 调用完成内容生成和分析任务。（课题任务1）
2. **开源项目宣传材料** — 结合本地知识库与网络数据生成的富文本，贴合官宣表达。（课题任务2）
3. **技术对比与路线判断** — 竞品技术优劣分析 + 智慧芽 AI 技术突破路线判断。（课题任务3+4）

**可用资源**：主流大模型 API + Vibe Coding 工具；Smart-Doc 文档解析 MCP；智慧芽公开论文与开源项目。

### 0.2 关键约束（写死，防跑偏）

- **知识组织走路线 B（LLM Wiki）**：资料策展成互链 Markdown 页面，agent 通过 read / grep / follow-link 使用。不建运行时图遍历引擎。
- **agent 策展，非 ETL**：脚本只做确定性搬运（解析/提取/建索引），wiki 页面、术语、风格、工作流由 Claude 策展撰写。
- **数据以公开为主**：论文/开源/官宣/竞品公开资料。企业记忆能力尽力而为，不硬吹。
- **主轴押"企业表达 + 术语"**：公开数据最能支撑风格与术语，主 Demo 是宣传稿（风格驱动）。
- **可溯源**：每个事实点绑回来源；无来源支撑标"待核实"，不编造。
- **竞品对比内容做真**：真查本地能力 + 真查网络竞品 + 真产出带证据矩阵；现场展示预置的真实结果，不做现场实时联网。
- **向量库**：已定采用。用于"语义模糊查"场景（见 §1.5），不是所有检索的必经路径。

### 0.3 开工前必须先确认的输入（缺则先问，不许瞎猜）

Claude 在动手前，先核对下列输入是否齐备。任一缺失，先向用户索取，不要用占位数据继续：

- [ ] **语料清单**：论文/开源项目/官宣文案的实际文件或 URL（公开为主）
- [ ] **主 Demo 目标项目**：宣传稿针对哪一个开源项目（点名一个）
- [ ] **竞品**：对比针对哪 1-2 个竞品
- [ ] **Smart-Doc**：`BASE_URL` 与调用凭证/鉴权方式
- [ ] **大模型 API**：生成用的 key 与模型名
- [ ] **向量库**：选型（哪个库、本地起还是云）、连接信息
- [ ] **术语真值来源**：可核对智慧芽标准术语的官网/产品界面/论文入口（术语表严禁凭印象编）

## 1. 公共引擎（三个产出共享的底层）

**核心原则**：脚本做确定性搬运，Claude 做认知策展。Skill 是给 AI 的能力包，不是数据库系统。

### 1.1 数据接入（脚本层）

只有确定性环节写成脚本。**理解、切分、判断噪声由 Claude 做**（读 `raw/` 后策展）。

**`tools/parse_docs.py`** — Smart-Doc 适配器
- 三端点：`/image/smart-doc`、`/pdf/smart-doc`、`/document/smart-doc`（Office 先转 PDF）。
- 请求参数：`detailed:true, markdown:true, ocr_backend:"auto", sign_url:false, reject_aspect_ratio:false`；PDF 加 `pdf_render_dpi:200, pdf_page_parallelism:4`。
- 响应是流式字符串块，`markdown:true` 时最后一块是拼接好的 Markdown → 落地到 `raw/<doc_id>.md`。
- **必须实现本地缓存**：以文件内容 hash 为 key，同文档不重复调用 API。
- **重要事实**：Smart-Doc 返回 Markdown + bounding box，**不返回结构化的 H1/H2/H3 树或表格行列 JSON**。标题层级和表格由后续从 Markdown 语法（`#`/`##`/表格管道符）自行解析。适配器只负责"拿到干净 Markdown + 缓存"。

**`tools/parse_code.py`** — 代码提取
- tree-sitter 解析开源项目代码；一个块 = 一个函数（签名 + 参数 + docstring），函数体丢弃。
- README/文档不走这里，走 parse_docs。

**`tools/build_index.py`** — 向量索引（语义查用，见 §1.5）
- 输入：切片 JSON；输出：写入向量库。**逐块灌入——向量库里每条记录 = 一个切片，不是整页文档。**
- embedding 模型：中英双语（如 BGE-M3 类，1024 维；**实际维度以选定模型为准，落地时确认**）。
- 离线批量运行，不常驻 GPU。
- **条件执行**：语料仅数十份且不做语义问答时，可暂不建索引（Claude 用 grep + 读页面覆盖）；语料上规模或需语义查时必建。

**Claude 做的（非脚本）**：读 `raw/` 的 Markdown → 判断并去除噪声（页眉页脚/导航/乱码）→ 按标题层级和语义段落切分（几百字/块，一块一个完整意思）→ 判定风格样本（官宣/技术博客整篇保留，不切）→ 为每块补元数据。

**切片元数据结构**（Claude 产出，供 build_index 和溯源用）：
```json
{
  "chunk_id": "eureka-readme-0007",
  "parent_doc_id": "eureka-readme",
  "source_type": "official|paper|code|doc|competitor",
  "project": "eureka",
  "usage": ["技术传播"],
  "title": "Eureka 产品说明",
  "year": 2024,
  "source_url": "https://...",
  "content": "……"
}
```

**验收**：给定一份 PDF/README，跑 `parse_docs.py` 得到 `raw/` 下干净 Markdown（二次调用命中缓存）；Claude 能据此产出带完整元数据的切片，风格样本另有整篇副本。

### 1.2 知识组织 · 路线 B（Claude 策展）

写成给 Claude 的策展指令，不是生成脚本。

**为什么是一个 skill，不是三个**：三个任务（宣传稿/竞品/路线）共用**同一个知识库、同一套术语、同一套人格**，只有"怎么做"不同。知识与人格全共享、仅工作流不同——这正是"一个 skill、多个 workflow"的场景，也贴课题任务1"设计本地知识库 Skill"（单数）。拆成三个 skill 会把 wiki/术语/风格复制三份，是维护负担。叙事上也更强：不是三个工具，是**一个懂智慧芽的数字员工**，会写宣传稿、会做竞品分析、会判断技术路线。（跨企业规模化时才分"通用底座 skill + 专属配置"两层，那是商业计划书 §6.2 的故事，非本次交付。）

**skill 内部两层**：
- **人格 + 知识**（共享，恒定）：`SOUL.md`（术语+风格）、`style-samples/`（整篇范文）、`wiki/`（知识页面）。
- **能力**（按任务加载）：`workflows/` 三个工作流，各是一套操作手册，也是 SKILL.md 对外暴露的三个能力入口。

**Skill 目录**（即 `patsnap-memory-skill/`，见 §0.4）。**wiki 页面类型**：
- `projects/` 每个开源项目一页
- `papers/` 每篇论文一页
- `technologies/` 每个核心技术能力一页
- `competitors/` 每个竞品一页
- `use-cases/` 每个 Demo 场景一页

**页面模板**（每页固定结构）：
```markdown
---
title: Eureka
type: project            # project|paper|technology|competitor|use-case
status: draft            # draft|confirmed（受控生长：AI 建初稿，人工确认后改 confirmed）
sources:                 # 溯源：本页每个事实的来源
  - {ref: eureka-readme, url: "https://..."}
updated: 2026-07-07
---

## 概述
（正文，每个事实点后标 [来源: parent_doc_id]）

## 相关
- [[technologies/知识图谱]]
- [[papers/xxx-2024]]

## 待核实
- （不确定的内容放这里，不写进正文当事实）
```

**策展指令（Claude 执行）**：
1. 读 §1.1 产出的切片与整篇文档。
2. 每个实体（项目/论文/技术/竞品/场景）建一页，用模板填写。
3. 事实点必须标 `[来源]`；不确定的进 `待核实` 段和 `error-book.md`，绝不写成事实。
4. 页面间建 `[[双向链接]]`：项目→用到的技术→相关论文→对标竞品。
5. 新建页 `status: draft`，人工确认后改 `confirmed`。

**受控生长机制**：AI 可建页、可提议链接，但重要页面和核心分类需人工确认；修正保留来源与版本；错误进 error-book。此机制是"企业版 LLM Wiki"区别于"个人第二大脑"的关键，也是答辩应对"企业能用吗"的答案。

**验收**：wiki 每类至少有真实页面（首版 15-25 页）；页面 `[[链接]]` 可跳转；每个事实有来源；不确定项在 error-book 而非正文。

### 1.3 三层注入（Claude 策展 + 运行时组装）

三层就是**三个由 Claude 撰写的 Markdown 文件**，内容是"规则/对照表"，不是数据。它们是 Claude 从素材提炼撰写的**产物**；运行时作为**恒定上下文**拼进每次生成的 prompt（每次都在，不检索）。下面给出每个文件的实际样例，未来的 Claude 照此格式撰写，不必猜。

术语层与风格层合并进一个 **`SOUL.md`**（企业的"语言人格"）。它俩体量都小、且都是恒定注入，合一个文件更简明、叙事更强（"注入企业之魂"）。但**术语在 SOUL.md 里保持独立一节**，好让"逐条核实"有清晰抓手——术语是客观对错（错了在专家评委前翻车），风格是主观贴合，两者对错性质不同，不能糊在一起。整篇范文因体量大，单独放 `style-samples/`。

#### `SOUL.md`（术语规范 + 风格规则）

**怎么做**：
- 术语节——读语料时记下出现的术语 → 逐条对照 §0.3 的术语真值来源（官网/产品界面/论文）核实 → 填表。**核实不了的不填，扔进 error-book**（严禁凭印象编）。
- 风格节——读 3-5 篇真实官宣 → 总结共同写法，写成**能照着改的动作**（不是抽象形容）。

```markdown
---
type: soul
updated: 2026-07-07
---

# SOUL — 智慧芽的语言人格

## 一、术语规范（客观，逐条核实过）
| 标准叫法 | 常见误用 | 推荐英文 | 来源 |
|---------|---------|---------|------|
| 专利同族 | 专利家族 | patent family | 官网/产品界面（待核实） |
| 智慧芽   | 智慧牙   | PatSnap        | 官网 |
| …（10-20 条，每条都核实过，来源列写清出处） |

## 二、表达风格（主观，对照样本改写）
1. 句式偏短，一句一个意思，避免从句套从句
2. 避免营销腔：不用"业界领先""颠覆性"这类形容词
3. 技术描述带具体数据：不用"大幅提升"，要用"提升 40%"
4. 主语习惯用"帮助企业/团队…"
5. …（每条都是可落地的改写动作）
   整篇范文见 style-samples/（体量大，此处只放规则）
```

#### `style-samples/`（整篇官宣范文，风格标尺）

**怎么做**：把 2-3 篇**整篇**真实官宣（新闻稿/技术博客）原文存入，一篇一个文件。**整篇不切片**——风格藏在谋篇、过渡、节奏里，切碎就没了。写稿时 AI 整篇读它，模仿句式与节奏（范文比规则更能传语感）。

```
style-samples/
├── 官网新闻稿-xxx发布.md
├── 技术博客-xxx.md
└── 海外开发者介绍-xxx.md
```

#### 工作流层 `workflows/*.md`（每个任务一套操作手册）

**怎么做**：把 §1.4 的生成自检流程，针对每个任务写成一份可照做的分步指令。三个任务各一份（见 §2-4）。样例（`launch-copy.md`）：

```markdown
---
type: workflow
task: 开源项目宣传稿生成
---

收到"给 X 项目写宣传稿"时，按此执行：
1. 读 wiki/projects/X.md 及其相关链接页，全量作为素材（§1.5：点名直读）
2. 联网查 X 项目的社区反响（§1.5：联网）
3. 加载 SOUL.md（术语+风格）
4. 生成初稿 → 术语自检 → 风格自检 → 事实自检 → 改写（§1.4）
5. 按固定结构输出：主稿 + 来源列表 + 待核实
```

#### 运行时组装（"恒定注入"是什么意思）

真要生成时，把上述三个 md 的内容 + 检索到的素材，**按固定顺序拼成一个大 prompt** 发给模型。术语表和风格指南**每次都拼进去**（所以叫"恒定注入"），保证每次生成都被同一套规则约束。

**Prompt 组装顺序（固定）**：
```
[工作流指令] 怎么做（本任务的分步流程）
+ [术语表] 按标准命名，禁用误写
+ [风格规则 + 整篇样本] 按谁的口吻写
+ [检索到的事实素材] 用什么料（每条带 source_url）
+ [用户任务] 具体要生成什么
```
检索结果只是"料"，三层注入是"手艺"。

### 1.4 生成与自检回路（Claude 工作流）

写成 Claude 的执行流程，不是黑盒 API。生成后必须自我批改再输出：

```
生成初稿
→ 术语自检：逐条对照 SOUL.md 术语节，改正误用命名
→ 风格自检：对照 SOUL.md 风格节与 style-samples/，改写不符处
→ 事实自检：每个关键事实点绑回 source_url；无来源支撑的标"待核实"
→ 改写终稿
```

**输出固定结构**（所有生成任务统一）：
```
## 主稿
（正文）

## 事实来源列表
- 事实点 → [来源: parent_doc_id / url]

## 待核实清单
- （无来源支撑或存疑的点，进 error-book）
```

**禁止条款**（写进每个生成 prompt）："只能使用提供的检索结果与注入内容，不得调用你的内部通用知识作答"。生成后检查输出是否含来源字段，缺失则重跑。

### 1.5 检索（三条线，向量库定位明确）

**先厘清知识的两种存在形式（关键，避免误解）**：

§1.1 产出的**切片（chunks）是唯一原料**。从切片出发，同一批内容变成两种形式：

```
raw Markdown
   ↓ Claude 切分 + 打元数据（§1.1）
切片 chunks（原料，一直存在，每块几百字带元数据）
   ├──→ Claude 策展重组 ──→ wiki 页面（.md，知识"正本"，给人/agent 直接读）
   └──→ build_index 逐块灌 ──→ 向量库（语义"索引"，每条记录 = 一个切片）
```

- **wiki 页面 = 正本**：Claude 把相关切片策展、重组、撰写成页面（如 `eureka.md`）。人能读、agent 能 read、能顺 `[[链接]]` 翻。
- **向量库 = 索引**：`build_index.py` 把切片**逐块**算 embedding 灌进去。**向量库里每条记录就是一个切片，不是整页。**
- 二者不是并列/二选一关系：向量库服务于"按语义找到相关切片"，wiki 页面服务于"直接读取策展好的完整知识"。

**检索不是三套系统，是一条分流规则。** 判据只有一个：**知不知道要哪一页**。像人查资料——知道在哪本书就直接翻，不知道才去搜，本地没有才上网：

```
if 用户点名了具体对象（项目名/文档名）:      # 例：给 Eureka 写宣传稿
    直接 read 整页 wiki/projects/<对象>.md + 它的 [[链接]]页 → 全量注入
                                            # 不走向量库：已知道要哪份，直接取比搜索更准、不漏段
else:                                       # 模糊问题，例："智慧芽做过图数据库吗"
    向量库语义召回相关切片（kNN + BM25，RRF 融合，元数据过滤收窄，取 topK）
                                            # 每条切片带 parent_doc_id/source_url，可溯源

if 需要外部信息（竞品资料 / AI 趋势）:        # 本地库根本没有的
    并联 tools/web_search.py 联网取公开信息
```

**三点说明**：

1. **点名直读 vs 语义召回，为什么分**：写宣传稿时你已明确要 Eureka 的**全部**资料，直接读整页不漏料；若改用向量召回只回 topK 个"最相似"切片，会丢掉该项目其余切片、还可能混入别项目的相似片段。**知道要哪份时，"取"比"搜"好。** 反之，"智慧芽做过X吗"这种你不知道答案散在哪，才需要向量库按语义把片段捞出来。

2. **召回的是切片，不是整页**：走向量库那条路，进出都是切片（每条带 `parent_doc_id/source_url` 溯源）；走点名直读那条路，读的是策展好的整页 md，不碰向量库。语料很小时，`else` 分支可用 grep + 读页面顶替；语料上规模或需真语义问答时，必用向量库。

3. **联网是并联支路，不是替代**：竞品对比、技术路线（课题②③④明写"结合网络数据"，硬要求）在本地取材之外并联联网。网络结果同样带 `source_url` 溯源；与本地冲突时以本地权威为准并标注；只取公开信息，不外传私域内容。**不做现场实时联网**——竞品/路线是准备阶段跑好、人工校对的结果（见 §0.2）。

**向量库选型**：用户定；万级切片以内单节点即可，不上集群。`build_index.py` 负责灌数据。

## 3. 产出②：开源项目宣传材料（主 Demo）〔课题任务2〕〔P0〕

**目标**：给点名的开源项目生成智慧芽风格的宣传稿，现场真跑通、真接大模型 API。

**切题锚点**：课题任务2、期望产出2"贴合官宣表达"。这是全案主 Demo（风格反差最直观，落在数据最强的表达轴）。

**workflow `workflows/launch-copy.md`**：
```
1. 精确取材（§1.5：点名直读）：读目标项目 wiki 页 + [[链接]]页 + 相关论文，全量注入
2. 网络补充（§1.5：联网）：项目在社区的反响/同类动态，补传播视角（呼应任务②"结合网络数据"）
```

## 4. 产出③：技术对比与路线判断 〔课题任务3+4〕〔P1〕

### 4.1 竞品技术对比（内容做真，现场展示预置结果）

**切题锚点**：课题任务3、期望产出3"竞品技术优劣分析"。

**workflow `workflows/competitor-analysis.md`**：
```

2. 网络取证（§1.5：联网）：查 1-2 个竞品公开资料
3. 按预设维度逐格填矩阵（数据能力/文档解析/专利检索/Agent能力/知识图谱/企业集成）

```
**输出**：对比矩阵表（维度 | 智慧芽表现 | 竞品表现 | 优势-不足 | 证据来源 | 置信度）+ 补强建议。
**重点**：不追求"智慧芽全赢"，追求判断有框架、有证据、有不确定性。

### 4.2 AI 技术突破路线（拔高）〔P2〕

**切题锚点**：课题任务4、期望产出3"AI 技术突破路线判断"。

**workflow `workflows/ai-roadmap.md`**：
```
1. 本地盘点现有能力（§1.5：点名直读/语义召回）
2. 网络取 AI 趋势要点（§1.5：联网）
3. 找现有能力与 AI 方向的结合点，输出 3 条候选路线
```
**输出**：3 条路线，每条含 业务价值/技术基础/落地难度/短中长期优先级/潜在风险。

**验收（4.1+4.2）**：矩阵每格有证据来源与置信度；路线有逻辑推导非空谈；均带溯源。

## 5. 评估：三方对比 〔P1〕

**目标**：把"感觉更好"变成"数字证明"，堵住"你们不就是个 RAG"的质疑。

**切题锚点**：证明个性化差异化，支撑三个产出的说服力。

**三方**：通用大模型 / 普通 RAG（=本方案关掉三层注入和自检）/ 专属方案（完整）。
**四项指标**：事实准确性、术语一致性、风格贴合度、改动量（反向）。
**方法**：`tools/eval_judge.py` 用另一个大模型当裁判（除"改动量"需人工），5-8 组样本。

**关键设计**：对照组"普通 RAG"= 本方案砍掉 §1.3 注入 + §1.4 自检后的形态 → **同一套管线关掉开关即可**，对比最干净，且证明"多出的分正是注入与自检带来的"。

**验收**：产出真实分数的对比表（非示意值）；能说清每项 5 分与 4 分差在哪。
