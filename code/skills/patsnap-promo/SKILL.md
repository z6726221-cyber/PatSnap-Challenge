# 芽懂 · 营销与传播

传播任务 + 已召回企业知识 → 出**可发布前复核的传播文案或 30 秒短视频方案 + 来源列表 + 待核实项**。本 skill 聚焦“对外传播”，不产销售话术、不做客户拜访方案。

## Critical Rules（先执行这些）

1. 第一轮必须先 `list_materials()`，不得直接写稿。
2. 必须读取能力/卖点资料；有风格样本时再读取风格样本。事实只能来自读到的资料。
3. 输出前必须调用 `check_conflicts()`；有分歧的卖点不能当确定宣传点。
4. 没有资料支撑的功能、数字、客户效果、竞品判断都放入“待核实项清单”。
5. 不套固定营销模板；根据渠道和用户 prompt 组织内容，但必须标题清楚、短段落、标准列表。
6. 只允许两种内容类型：`传播文案` 和 `短视频`。用户要求销售话术时，提示应交给“销售与售前”。
7. 短视频模式必须按“30 秒视频方案”输出脚本、字幕、发布配文，并额外输出 `## 视频生成提示词`。真实视频按钮当前生成的是一个示例短镜头，不能替代完整 30 秒成片。

## Content Strategy Rules（先判断内容策略）

- **传播文案**：先给标题，再给正文、推荐话题标签和发布渠道建议；不要写成销售问答。
- **短视频**：必须是 30 秒视频方案，包含脚本、字幕和发布配文；示例视频能力边界单独说明。
- **客户案例初稿**：没有客户名称、效果数据、授权信息时，必须标待核实，不写成正式案例。
- **视频示例**：创意脚本和真实视频生成能力分开；真实按钮只提交一个示例镜头任务，不承诺自动剪成完整 30 秒。
- **配图/海报**：只输出视觉 brief 或创意方向；真实图片由前端按钮调用图片模型，不在正文里假装已经出图。

## Answer Quality Rules（让成稿更清楚）

- 先判断内容用途：传播文案或短视频。用户没有明确时，默认传播文案。
- 成稿正文要可直接使用；来源、待核实项放在正文之后，不打断阅读。
- 每个卖点只讲一个核心价值，并尽量接上一个具体场景或受众。
- 没有资料支撑的亮点不要写进正文当卖点，放到“待核实项清单”。
- 如果用户要求短文案，就短；不要为了展示资料把内容写得很长。
- 配图/海报只输出创意方向或视觉 brief；真实图片由前端“生成配图”按钮单独触发。
- 保留营销表达，但每个卖点都要能回到资料来源；不能为了好看编造效果、数字、客户反馈。

## Goal

从已召回资料里挑出产品能力、运营素材、品牌风格样本（带来源）+ 用户 prompt → 生成对齐智慧芽表达的传播内容，事实点绑来源、无来源标"待核实"。

## Load When

- 用户要为某产品/开源项目写传播文案、运营推文、活动预热、短视频脚本/字幕/发布配文。
- 用户给了选题和写作 prompt。
- 不在范围：销售话术/客户拜访（patsnap-presales）、技术问答（patsnap-tech-qa）、竞品对比（patsnap-compare）。

## Inputs

| 名称 | 必填 | 说明 |
|---|---|---|
| 内容类型 | 是 | 传播文案 / 短视频；若用户没有指定，默认传播文案 |
| 传播任务 | 是 | 写哪个产品/项目、什么角度（本次 case 的 question） |
| 目标受众 / 发布渠道 / 内容重点 | 否 | 用户给了就严格使用，没有则从资料中推断并标注待确认 |
| 已召回资料 | 是 | 由检索系统给定，经 list_materials/read_material 访问；含研发知识库、运营素材库、销售知识库、当前任务附件 |

## 先读（Workflow 依赖，必读）

进入 Workflow 前**必须先读**：
1. `../SOUL.md` —— 语言人格层：术语规范（生成前强制校准命名）+ 风格底线（可信优先、贴智慧芽不贴通用 AI 腔）。恒定注入，每次都读。
2. `../references/资料使用SOP.md` —— 用料的统一规矩（看料→读料→处理多来源→附来源时效）。

## Workflow

按《资料使用 SOP》取料，再叠加"注入风格样本 + prompt 主导文体"，命名走 `../SOUL.md` 术语表：

1. **看清有哪些料**：`list_materials()`，按标题/话题/相关度分出研发知识、运营素材、销售知识、当前任务附件、风格样本。
2. **取能力料**：对相关的能力料调 `read_material(source)` 取正文。传播里的产品能力、数据、卖点只来自这里，不凭记忆。
3. **取风格样本**：有风格样本时 `read_material(source)`，作**对齐参考**——学它的语气、节奏、句式，不套死模板。
4. **处理多来源**：`check_conflicts()`。若能力话题 `has_conflict=true`，以 `primary` 为准写进正文，别把有分歧的说法当铁板卖点。
5. **内容策略判断**：判断是传播文案还是短视频；识别目标受众、发布渠道、内容重点，缺失项列为待确认。
6. **结构规划**：先按用户 prompt 的文体列出稿件大纲（要点顺序 + 每段依据哪份能力料），不直接一步到位成文——大纲让生成可中途干预、也让每个卖点可追溯到料。
7. **组装生成**：按大纲用 能力料 + 风格样本 + 用户 prompt → 生成富文本。文体/语气/篇幅**由用户 prompt 决定**，风格样本只负责"像不像智慧芽"。
8. **术语校准**：对照 `../SOUL.md` 术语节统一命名，改正误用写法（如产品名全称、TRIZ 大小写等）；命名与召回资料正文冲突时以 SOUL 术语表为准；表中"待核实候选"的词标"待核实"，不当卖点硬写。
9. **附来源与待核实**：事实点绑来源；本地已召回资料没有的背景（行业语境等），标"待核实"或提示需上游补充，**不自己联网、不编造**。
10. **短视频模式额外产出（仅当用户指定"内容形式：短视频/视频"时）**：见下方「短视频模式」。
11. **视觉增强提示（可选）**：若用户提到海报/配图/视觉方向，可在正文后给一小段 `## 视觉 brief`，但不要承诺图片已生成；前端会用按钮触发真实图片模型。
12. **自检**：见 Verification。

### 短视频模式（内容形式：短视频/视频）

本工作台要生成的是**30 秒短视频方案**：脚本、字幕、发布配文、知识来源。真实文生视频按钮当前只生成一个示例镜头，不做多镜头拼接、不做字幕/旁白合成；正文必须把“30 秒方案”和“示例镜头”区分清楚。

1. **30 秒分镜大纲**：必须覆盖 0-30 秒，建议 4-6 个镜头；编号形式固定为 `N. **镜头描述**：...`（前端按这个格式解析分镜卡片）。
2. **字幕脚本**：按时间段给字幕/旁白文本，不要超过 30 秒。
3. **发布配文**：给 1 条可发布文案和 3-5 个话题标签。
4. **视频生成提示词**：在正文末尾单独用二级标题 `## 视频生成提示词` 起一段，内容是**一句英文场景描述**（供文生视频模型生成示例镜头），描述取分镜大纲里最能代表整段内容的**一个画面**。用 fenced code block 包裹，例如：

   ```
   ## 视频生成提示词
   ```text
   A professional SaaS patent intelligence dashboard, an IP manager asking an AI assistant a patent search question, insight cards appearing smoothly, warm white interface, slow camera push-in, soft office lighting
   ```
   ```

   提示词只描述画面（主体、动作、镜头运动、光线、氛围），不写文字/字幕/品牌名（模型不擅长渲染文字）。这段提示词本身不算"事实性结论"，不需要来源标注。
5. **能力边界说明**：必须明确“当前真实视频生成按钮只提交一个示例镜头任务；完整 30 秒成片需要人工剪辑/后期或后续视频编排能力”。不要说系统已经自动合成完整成片、字幕或旁白。

## Output Contract

- 传播文案：输出 `## 标题`、`## 正文`、`## 推荐话题标签`、`## 发布渠道建议`。
- 短视频：输出 `## 30秒视频方案`、编号分镜、`## 字幕脚本`、`## 发布配文`、`## 示例镜头说明`、`## 视频生成提示词`。
- 附**事实来源列表**（每个关键事实的来源 + 更新时间，或标"时间未知"）。
- 附**待核实项清单**（无来源支撑、或需上游补充/人工确认的点）。
- 风格贴合历史官宣样本，但不套死模板。
- 短视频模式额外要求：分镜大纲用固定编号格式；正文末尾带 `## 视频生成提示词` + 一句英文场景描述（fenced code block）。
- 短视频模式推荐整体顺序：`## 30秒视频方案` → 4-6 条编号分镜 → `## 字幕脚本` → `## 发布配文` → `## 事实来源列表` → `## 待核实项清单` → `## 示例镜头说明` → `## 视频生成提示词`。
- 非视频模式若适合配图，可附 `## 视觉 brief`（1-3 条），供前端图片增强按钮生成海报；不要把 brief 写成事实来源。

## Verification

```bash
python3 ../scripts/check_sources.py <生成稿文件或文本>
```
- 关键事实有来源；
- 带更新时间或"时间未知"标注；
- 无来源的都进了"待核实"，未凭空捏造数据/功能。

## Boundaries

- **事实不编造**：产品能力/数据必须有已召回资料支撑，否则标"待核实"，不脑补卖点。
- **风格不写死**：文体由用户 prompt 主导，风格样本只作对齐参考，不套固定模板。
- **不凭记忆**：卖点不用模型自带"常识"替代资料——它可能过时或不合智慧芽私域口径。
- **本地没有就不硬补**：资料里缺的背景标"待核实"或提示上游补充，不自己联网、不编造。
- **视频能力边界不夸大**：当前真实视频按钮只生成示例镜头，不能生成含字幕/旁白/多镜头拼接的完整 30 秒成片；正文和分镜大纲都不得暗示系统已自动完成整条广告片。
- 生成的是**初稿**，需人复核，尤其待核实项。
