# Plan A · 基于 OpenViking 的技术方案

> 面向运营销售的私域技术知识库 · 基于 OpenViking 上下文数据库
>
> 本文是课题2的**主方案(Plan A)**。一句话:**检索与存储引擎不自研,直接用火山引擎开源的 OpenViking(AI 智能体上下文数据库)当底座**。它自带文档解析、分层组织、目录递归检索、版本控制——于是我们的工作**从"造引擎"变成"喂对数据 + 补它的短板 + 做场景逻辑 + 接前端"**,一周内就能跑通完整 Demo。

## 一、产品定位:先想清楚给谁用

技术选择全由"用户是谁"决定。这个产品的用户**不懂技术**,方案的每个取舍都顺着这一点。选 OpenViking 也是这个逻辑——把复杂的检索工程交给成熟开源件,我们专注贴合运营销售的场景。

- **用户:运营 / 销售 / 市场 / 内容策划。** 他们不懂 TRIZ、不懂专利技术细节,平常遇技术问题都问研发、研发又讲不明白、响应慢。他们要的是用大白话问、拿到能直接用的答案。
- **最痛的点:技术口径不一致。** 不同销售对同一技术能力说法不同,客户面前容易露馅。知识库的头号价值是**统一口径**——同一问题,谁问都是同一个有出处、有时效的标准答案。
- **核心是"存"与"召回"。** 回答风格、生成什么报告,都由用户自己写 prompt 决定。所以不在生成花样上堆工程,重心是数据怎么存、怎么精准召回——这正是 OpenViking 擅长的。
- **溯源是刚需。** "统一口径"要经得起客户追问。每个答案必须带来源。OpenViking 每条结果都返回 `viking://` URI,天然可溯源。
- **内外数据结合。** 宣传智慧芽必然有横向对比。系统要结合本地能力 + 外部竞品信息——需要时联网搜,或用户抛一个竞品网址,让系统说清我方优势。

> **定位:能力是主角,数据是并列样例。** 产品是**通用的私域知识能力平台**——**任意**智慧芽私域资料进来都能建库、问答、生成宣传、竞品对比、路线判断。本方案描述的是这套**与具体数据无关的能力**,不锁定"主样例";落地时挑 1-2 个数据集把闭环跑透即可,具体选哪个是 Demo/企划书的事。(对齐课题定向 v2:v2 设想的"向量检索+RAG",本方案用 OpenViking 具体化并升级,核心判断不变。五模块参考结构——方法论/产品能力/宣传语料/竞品/技术路线——在第四、五节展开。)

## 二、为何选 OpenViking

OpenViking 是火山引擎开源的"**AI 智能体上下文数据库**"(非传统向量库)。它主动区别于扁平向量 RAG:用文件系统范式组织知识、自动分层、目录递归检索。选它的理由很实际——把我们本来要自研一周的引擎,变成开箱即用,省下的时间投到贴合场景的部分。

| 能力 | 自研向量库要做的 | OpenViking 现成给的 |
|---|---|---|
| 文档解析 | 接 OCR、写各格式适配器 | 自带解析 PDF/Word/PPT/Excel/网页/代码库/飞书文档 |
| 分片 + 分层 | 自己切片、拼上下文前缀 | 导入即自动生成 L0 摘要 / L1 概览 / L2 详情三层 |
| 索引 + 检索 | 向量库 + 混合召回 + rerank 全都自己搭 | 意图分析 + 目录递归检索 + rerank 内置 |
| 版本 / 更新 | 自己设计版本字段、去重逻辑 | git 快照能力(commit/restore/show)+ watch 定时增量更新 |
| 接入 | 自己写 API | 本身是 MCP server + HTTP API + Python SDK |
| 可观察 | 自己做日志 | 检索轨迹可视化(目录浏览路径可回看) |

**一句话:** OpenViking 把"数据接入 → 分层组织 → 检索 → 版本管理 → 接入"整条链路都做了。我们的工作从"造引擎"变成"喂对数据 + 补它的短板(专名召回)+ 做场景逻辑(竞品对比)+ 接前端"。这也让一周做出完整 Demo 变得现实。

**⚠️ 但要如实认识它的两个边界(详见第十四节):**

1. **开源版默认不是真 BM25**——真正的 BM25 混合检索要接入火山商业云 VikingDB 才启用(源码 `grep_config.py` 注释明确写"vikingdb bm25 recall when available");开源版默认走本地正则 `grep`(`viking_fs.grep()`,是它自带的 API,不是我们补的能力)。中文专名(Eureka/TRIZ)纯向量可能召不回,我方的工程贡献是**编排层判断何时该触发 grep、如何与语义检索结果融合**,不是"造了 grep 这个能力"。
2. 主项目是 **AGPLv3** 协议,商业化前**必须**由法务评估(不是"建议咨询",AGPL 的网络交互条款可能使"HTTP API 黑盒调用"这层隔离仍不够,见第十四节)。

## 三、整体架构:产品 ↔ HTTP API ↔ OpenViking 独立服务

架构的关键是**边界清晰**:OpenViking 作为独立服务跑在后面,我们的产品只通过 HTTP API 调它。这既是工程解耦,也是 AGPL 合规的护栏——不把它的代码编进我们的产品。

```
运营/销售(浏览器)
  → 我方产品前端 + 编排后端
  → OpenViking HTTP API
  → OpenViking 服务(检索/存储/版本)
```

我方"编排后端"负责 OpenViking 管不了的场景逻辑:按五模块圈定检索范围、专名 grep 兜底、竞品对比的查询分解与维度对齐、外部网页实时抓取、把检索结果 + 用户 prompt 组装后调大模型生成、给答案附来源与时效。

## 四、数据接入:OpenViking 自带解析为主,关键件用 smart-doc 兼容

数据进库靠 `ov add-resource`(支持 URL、本地文件、整个文件夹、Git 仓库)。OpenViking 自带解析,导入后自动分层 + 向量化,不用我们自己切片。

**1. 主力:OpenViking 自带解析。** Word/PDF/Excel/网页/代码库/飞书文档直接 `ov add-resource` 导入。代码库支持 Git URL、本地 .git、.zip,遵循 .gitignore;网页支持同域递归爬取。导入后自动构建 `viking://` 资源树并生成 L0/L1/L2 分层。

```bash
# 导入一个开源项目(产品能力模块)
ov add-resource https://github.com/xxx/eureka.git \
  --to viking://resources/products/eureka --watch-interval 60 --wait

# 导入本地资料文件夹(方法论/宣传语料模块)
ov add-resource ./triz_docs/ --reason "TRIZ 基础资料" --wait
```

**2. 兼容:关键件用 smart-doc 先解析。** OpenViking 自带解析对普通文档够用。但**图多、信息在图里的 PPT,以及扫描件**,自动解析可能丢信息。对这类关键件,先用智慧芽内部的 Smart-Doc API 解析成高质量 Markdown(版面分析 + OCR),再把 Markdown 喂给 OpenViking。这是一条兜底通道,不是主力——大多数资料直接进 OpenViking。

**3. 按五模块组织目录。** 导入时用 `--to viking://resources/<模块>/<项目>` 把资料归到五个模块目录下(方法论 / 产品能力 / 宣传语料 / 竞品 / 技术路线)。**目录结构就是后面检索圈定范围的抓手**——OpenViking 的目录递归检索靠它。竞品资料单独放 `resources/competitors/`,便于对比时分开取。

> **力气分配变了**:自研方案里最重的"解析 + 切片 + 建索引",OpenViking 全包。我方在数据接入上只做两件事:关键件走 smart-doc 提质、按模块规划目录。省下的力气投到检索补强和场景逻辑。

## 五、知识组织:viking:// 文件系统 + 三层加载

OpenViking 不把知识存成扁平切片,而是组织成一个虚拟文件系统 `viking://`,每个资源有唯一 URI、有层级、有分层摘要。理解这套范式,才能用好它、也才能在答辩讲清"为什么比裸 RAG 强"。

- **文件系统范式。** 记忆、资源、技能统一映射到 `viking://` 虚拟目录,每个条目一个 URI。系统能像开发者一样用 `ls / find / grep` 精确定位,把"模糊语义匹配"变成"可追踪的文件操作"。对我们的价值:目录即模块,天然对应五模块结构和权限范围。
- **L0 / L1 / L2 三层加载。** 每个条目自动生成三层:L0 摘要(约 100 字,快速判相关)、L1 概览(核心信息)、L2 详情(完整原文,按需加载)。检索先看 L0/L1 定位,真正要用才拉 L2。对我们的价值:省 token、也让"统一口径"能先给结论再给细节。

```
viking://resources/
├── methodology/        # 方法论模块(TRIZ 等)
├── products/eureka/    # 产品能力模块(1-2 个开源项目做透)
│   ├── .abstract       # L0 摘要
│   ├── .overview       # L1 概览
│   └── docs/ src/ ...  # L2 详情
├── style/              # 宣传风格语料(整篇,权重低)
├── competitors/        # 竞品信息(与本地同结构,便于对比)
└── roadmap/            # 技术路线资料
```

> **这正是它像 Wiki 又不是 Wiki 的地方**:它用"目录层级 + 分层摘要"承载结构(类似 Plan B 的 Wiki 思想),但结构由导入时自动生成,不用人工建页维护链接。所以 Plan A 用 OpenViking,本质是"用自动化的结构化组织,换掉 Plan B 需要人工维护的 Wiki"。

## 六、检索:四步 SOP —— 引擎理解口语 + 我方圈范围与专名兜底

检索是所有读取类场景(问答/对比/宣传取料)的公共动作。我方把它固化成一套**四步 SOP**:前两步里"读懂口语"交给 OpenViking 引擎(它的 `ov search` 自带意图分析),我方只补引擎做不了、且是我方核心价值的两件事——**按五模块圈定范围**和**专名精确兜底**。

> **命令核准说明**:范围限定参数是 **`--uri`**(不是早期草稿里的 `target_uri`/`--target`);口语检索优先用 **`ov search`**(带意图分析、查询扩展),`ov find` 是纯向量、留给已精确化的专名查询。这些命令名对照 OpenViking 本地文档核准;本节描述的是**接入真引擎后**的检索链路(当前落地为生成侧闭环,检索由上游模拟,见第十一节落地状态)。

**① 口语检索(引擎理解意图)。** 直接把运营的口语问题交给 `ov search`——它内置意图分析 + 查询扩展 + 会话上下文,已经把"读懂口语、改写成检索友好表述"这件事做了。**所以我方不再自己先过一遍小模型改写**(那是早期设想,与"不重复造轮子"冲突,已砍掉)。每条结果返回 `viking://` URI 作来源。

**② 按五模块圈范围(我方判断,引擎不懂我们的目录结构)。** 引擎的通用意图分析不知道"芽懂"的五模块结构,圈不准。我方编排层据问题判断属哪个模块,用 **`--uri`** 收窄(如 `--uri viking://resources/products/eureka`)。**范围圈对了,后面就对了一大半。**

**③ 专名兜底(我方判断,引擎不认智慧芽专名)。** `grep`(正则精确匹配)是 OpenViking 自带 API,开源版默认不接真 BM25(那要接火山商业云 VikingDB),中文专名(Eureka/TRIZ)纯向量可能召不回。我方编排层做**判断逻辑**:query 含专名词典里的词时,并行跑一次 `ov grep <专名> --uri <范围> --ignore-case`,与 `search` 的语义结果融合排序。工程贡献在"何时触发、怎么融合"。

**④ 附来源 + 时效(产物,交给校验脚本兜底)。** 每条采用的结果取其 `uri` 字段作来源;更新时间从资源元数据/`ov stat` 取(不假设检索结果直接带时间)。这一步是机械可验证的产物,由共享脚本 `check_sources.py` 卡死,不靠 agent 自觉(见第十一节)。

```
口语 query
 → ① ov search(引擎理解意图)  --uri <②我方按五模块圈的范围>
 → ③ 含专名并行 ov grep 兜底,融合去重
 → ④ 附来源(uri 字段)+ 时效(ov stat)
 → 喂大模型
```

### 最难的场景 · 对比类提问(我方)

销售天天问对比:"我们 Eureka 的多语言比竞品X强在哪?"直接检索会偏向一边。编排层做**查询分解 + 多源分别召回 + 按维度对齐**——这段逻辑 OpenViking 不管,是我方核心工程。

```
用户问:"Eureka 多语言 vs 竞品A 强在哪?"
我方编排层:
  ① 拆解 → {维度:多语言, 我方:eureka, 竞品:[竞品A], 时间:最新}
  ② 路A: ov search ... --uri viking://resources/products/eureka    → 本地
     路B: ov search ... --uri viking://resources/competitors/竞品A → 竞品
          (竞品库没有 → 触发实时抓取,见第八节)
  ③ 按"多语言"维度对齐两边,附各自来源,喂模型生成对比
```

> **没有这个分解,对比类回答一定差。** 这恰是产品核心卖点(内外结合),所以是我方必做的工程,不能指望 OpenViking 内置检索直接搞定。

## 七、时间线 / 版本:直接用 OpenViking 原生能力

"资料会更新、口径要用最新版"是本产品的硬需求。好消息:**OpenViking 原生支持**,我们几乎不用自研——这是选它的一个大加分项。

- **git 快照能力。** OpenViking 设计了基于 git(gitoxide in-process)的账号级快照,三个原语 **`commit / restore / show`**,每个 account 一个逻辑仓库,历史不可改写(restore 生成新提交而非覆盖)。每次资料更新可留一个可追溯的版本点。
  > ⚠️ **核准说明**:这是 OpenViking 版本控制**设计文档**里的原语(可能以内部方法 `VikingFS.commit/restore/show` 形式提供),**未必是已稳定发布的 `ov` CLI 命令**。答辩引用时表述为"OpenViking 的账号级快照能力(commit/restore/show)",并注明依赖版本,不要说成现成命令。早期草稿写的"commit/log/restore"有误——是 `show` 不是 `log`。
- **watch 自动增量更新。** 导入时设 `--watch-interval`,OpenViking 会定时从 Git 仓库/网页源自动检测更新并同步。资料一变,库自动跟进,不用人工重导。也可 `trigger_watch` 手动立即刷新。

> **与"统一口径"的联动(答辩亮点)**:默认检索命中的就是当前最新版;需要时可用快照回看历史口径。我方在输出层再补一步——把答案附上来源 URI + 该资料的更新时间,让销售看到"答案 + 出处 + 截至几月的口径"。统一口径 = 统一内容 + 统一时效,而底层版本管理由 OpenViking 兜底。

### 七之二、更难的问题:不同资源互相矛盾,怎么办

上面说的是"同一份资源自己更新了"(OpenViking 原生管)。但还有一种更难的情况:**两份不同来源的资源,对同一件事说法不一样**——比如内部 PPT(2025-03)说"支持3种语言",官网(2026-06)说"支持5种语言"。这两份不是新旧版本关系(不是同一个 URI 的历史),是**两个独立资源打架**,OpenViking 的 git 快照 / watch 管不到这层,因为它按资源(URI)做版本,不按事实做仲裁。

**查证过:OpenViking 没有现成机制能直接拿来用。** 它内部确实有一套讲究的合并逻辑(`session/memory/merge_op/`,支持 patch/replace/sum/immutable 四种合并策略,还带 `confidence` 参数),但这套东西是给**长对话 User Memory / Agent Memory** 用的(官方 LOCOMO 评测测的就是这类场景)——解决"agent 记住的用户信息,该怎么随新对话更新",不是给 `viking://resources/` 这层**静态知识资源**设计的。我们要处理的问题落在 resources 层,这层要自己做编排逻辑。

**业界怎么处理这题(不是我们特例,是个公认的难题)。** 让 AI 自动判断"两句话语义上矛不矛盾、谁对谁错",目前没有稳定方案,主流产品都不硬解这题,退而求其次用两条更笨但可靠的规则:

1. **时效 + 权威性优先(决定"谁当主答案")**:同权威级别下选更新的;不同权威级别按来源分级(官网/产品界面 > 内部权威文档 > 论文 > 二手转述),级别高的优先,不是唯"越新越对"。这一步负责排出"该信哪个"。
2. **曝光而非静默覆盖(决定"输家怎么处理",Guru 等企业知识库产品的通用做法)**:排序选出的赢家当主答案,但**不删除、不隐藏**输的那份——回答里附一句"另有资料记录不同,来源 XX、更新于 XX",让销售自己知道曾有分歧、看到依据。这一步不是取代第1步,是给第1步的判断结果加一层"可核查、不装懂"的呈现。

两步合起来才是完整机制:**第1步决定说什么,第2步决定"没被选中的信息要不要一起亮出来"**。只做第1步,销售看到的是一个自信但可能过时的答案,客户深挖一句就露馅;两步都做,销售能看到"当前口径是什么 + 曾经的说法是什么 + 为什么现在信这个",这才是真正经得住追问的"统一口径"。

**具体落地(不需要改 OpenViking 源码,编排层做):**

```
建库规范 skill 归档新资料时:
  ① 在同模块内做一次话题重复检测(用 ov find/grep 圈定同话题已有资源,
     不是语义冲突判断——只是"话题上有没有已存在的资源")
  ② 若有:两份都保留,不覆盖;按权威性规则打标签(我方定义的字段,
     不依赖 OpenViking 现成能力),记一条"同话题多来源"关联

技术问答 skill 检索命中时:
  ③ 若候选结果里有"同话题多来源"标记:
     - 按权威性+时效选一个作主答案
     - 输出时附上"另有资料表述不同"的提示 + 另一份的来源与时间
  ④ 定期(而非实时自动化)人工复核"同话题多来源"列表,
     确认要不要更新权威性标签或直接合并成一份——知识库范围窄(1-2个技术领域),
     人工复核的工作量可控,不需要建复杂的自动提醒系统
```

> **与 Plan B 的一致性**:这套"权威性裁决 + 曝光而非静默覆盖"的思路,和 Plan B(`error-book.md` + 冲突裁决规则)是同一套心智模型,只是 Plan A 里落地成"编排层判断 + OpenViking 资源标记",不是"Wiki 页面的 frontmatter 字段"。两个方案在这题上判断一致,答辩时可以讲成"两条技术路线,同一个治理原则"。

## 八、内外数据结合:预存竞品(主 Demo)+ 实时抓取(亮点)

销售宣传必有横向对比,外部竞品数据是刚需。两条腿:主 Demo 用预存竞品资料保稳,另做"抛竞品网址实时抓取"作亮点。

- **预存竞品(主 Demo,稳)。** 提前把主要竞品资料 `add-resource` 到 `viking://resources/competitors/`,与本地同结构。现场不联网,检索时用 `--uri` 把本地能力与竞品分开取,再对齐。
- **实时抓取竞品 URL(亮点,我方)。** 用户抛一个竞品网址,我方编排层抓取该页 → 去噪 → 按对比维度抽取 → 与本地对齐。也可即时 `add-resource` 该 URL 让 OpenViking 解析入库再检索。演示最有冲击力。

> **抓来的网页不可信、格式乱,不能让模型自由发挥。** 流程要结构化:抓取 → 去噪 → 按"我方已知的对比维度"逐项抽取竞品信息(维度由本地能力驱动)→ 逐维度对齐。缺失维度明确标"未在该页找到",不猜。外部数据同样附 `source_url + 抓取时间`,冲突时以本地权威为准并标注。联网只取公开信息,不外传私域内容。

## 九、宣传材料生成(官方任务2)

这是官方**三项交付物之一**(为开源项目生成富文本宣传材料)。方案里生成端**刻意做薄**——风格与文体由用户 prompt 控制,不写死模板。但"做薄"不等于"不做":要讲清宣传材料怎么走完整条链路产出,并守住"可信、贴官宣、带来源"三条底线。

```
用户选题 + 写 prompt
 → 检索本地(项目能力,带 URI 来源) + 联网补背景
 → 组装:料 + 风格样本 + 用户 prompt
 → 生成富文本 + 附来源列表 + 待核实项
```

- **料从哪来**:项目能力、技术亮点从本地知识库检索(带 URI 来源);项目背景、行业语境可联网补。事实点绑来源,无来源的标"待核实",不让模型编。
- **风格怎么贴**:不写死模板。检索 `style/` 模块里的历史官宣文案作**风格样本**一并注入,让生成对齐智慧芽表达;具体文体(公众号/话术/推文)由用户 prompt 指定。
- **产出形态**:公众号文章、产品介绍、销售话术、活动推文、客户案例初稿等富文本。每篇附事实来源列表 + 待核实项,保证可信、可追。

> **为什么生成端薄仍合理**:课题核心是"数据怎么存、怎么召回",风格由用户 prompt 控制是 v2 明确的技术判断。所以我们不在生成端堆模板工程,而是保证检索给对料、风格样本注入到位、来源可溯——生成质量的地基在检索和料,不在花哨的生成逻辑。

## 十、技术路线判断(官方任务4)

官方四大任务之一(分析智慧芽在 AI 浪潮中的技术突破路线)。v2 特别强调它"虽难但出彩,别丢"。做法:**把"自身现有能力"和"外部 AI 趋势"两半拼起来,找结合点**,人做判断、系统供弹药。

- **内半:自身能力盘点。** 从本地知识库的产品能力模块提取智慧芽现有技术能力清单(什么能力、成熟度、覆盖场景)。这半是私域数据的强项,可靠。
- **外半:AI 趋势。** 联网检索当下 AI/大模型趋势(能力边界、行业方向)。这半本地没有,靠外部数据 + 人的判断。明确标注这是外部信息、可能过时。

```
现有能力盘点(本地) × AI 趋势(联网) → 找结合点 → 路线判断(人复核)
```

> **诚实边界**:路线判断是**辅助判断,不是自动结论**。系统的价值是把"现有能力"和"AI 趋势"两半材料备齐、给出候选结合点,最终判断由人做、并标注哪些是数据支撑、哪些是推测。这题难在结论的可信度,所以宁可给"有依据的候选方向",不要给"看似确定的假结论"。

## 十一、Skill 交付形态

官方期望产出第一项白纸黑字是"**本地知识库 Skill**(非泛泛的向量库,要能被 AI 调用的 Skill/MCP 形态)"。用了 OpenViking 后,这点**更要讲清边界**——否则会被质疑"那 Skill 是你做的还是 OpenViking 做的?"。

### 先看清三层:skill 既不是知识、也不是检索,是中间的"操作规矩层"

最容易误解的两点:**①知识不在 skill 里,在 OpenViking 里**(`viking://resources/` 下的资料才是知识,skill 里只有"怎么用库的规矩"、不抄知识);**②OpenViking 既是存储又是检索**,不只是检索层。

```
① Agent 层(大脑)     AI 模型(Claude / 产品后端)—— 读 skill、做决策
        ↓ 照着 skill 的规矩办事
② Skill 层(操作规矩)  我方写的 SKILL.md —— "智慧芽场景下怎么用库" 的 SOP ← 我方交付物
        ↓ 调用 ov find / ov grep
③ OpenViking 层        存知识(viking://resources/)+ 检索(find/grep/rerank)
```

- **Agent 层 · 大脑(用现成的)**:AI 模型读 skill、做决策、照规矩办事。不是我们造的。
- **Skill 层 · 操作规矩 SOP(← 我方交付物)**:我们写的 `SKILL.md`,里面没有知识,只有编排规矩,调用下层的 `ov find`/`ov grep`。这层才是我们的技术贡献。
- **OpenViking 层 · 存知识 + 检索(用现成的)**:智慧芽资料(知识)存这里,检索也在这里。存储和检索都归它管。

> **图书馆比喻,一句话记住**:OpenViking 是**图书馆**(既存书、又帮你找书),skill 是给管理员的**工作手册**(遇到这类问题去哪个书架、怎么核对出处)。手册可以放在前台抽屉里(skill 能用 `ov skills add` 安装进 OpenViking 托管),**但执行手册的是管理员(Agent),不是书架(检索引擎)**——skill 没有变成引擎的一部分。所以"嵌入"只是托管,不是被引擎执行。

> **一句话答清评委**:"我们交付的 Skill = 在 OpenViking 引擎之上,封装的一套面向智慧芽私域知识的场景能力(建库规范 + 检索编排 + 竞品对比 + 溯源)。引擎(解析/分层/检索/版本)用成熟开源件,场景化封装是我们的工程。"这样既满足官方"Skill/MCP 形态"的要求,又说清了技术贡献。

### Skill 到底是什么形态(破除"skill=代码模块"的误解)

对齐 OpenViking 官方 skill 示例(`ov-add-paper`、`ov-resources`):一个 skill 就是**一个文件夹**,核心是一份给 AI 看的**自然语言操作说明书(SOP)**,不是一坨程序——它的价值是"教会 AI 按规矩使用知识库",检索本身归 OpenViking 引擎。**没有传统代码,规矩即工程。**

- **SKILL.md(核心)**:YAML 头(name / description 触发条件 / 兼容性 / allowed-tools)+ 正文(Goal / Inputs / Workflow / Boundaries / Verification)。用**规矩**把 OpenViking 原子能力编排成"智慧芽场景下的正确用法"。
- **references/(可选)**:参考文档,AI 干活前先读(如建库规范、对比维度清单)。
- **scripts/(可选)**:少量确定性校验脚本(如导入格式校验),不是主体。

### 我方要做的一组 Skill(对应四大官方任务)

| Skill | 做什么 | 对应任务 |
|---|---|---|
| 建库规范 skill | 教 AI 把资料按五模块归到正确的 `viking://` 目录;关键件先走 smart-doc;归档时做话题重复检测,同话题多来源打权威性标签(见第七之二节) | 任务1 |
| 技术问答 skill | 口语查询 → `ov search` + 专名 `grep` 兜底融合 → 附来源与时效;命中"同话题多来源"标记时,主答案之外附注另一说法。核心是统一口径 | 任务1 应用 |
| 竞品对比 skill | 对比问题拆解 → 本地与竞品分别召回 → 按维度对齐 → 标注冲突与缺失 | 任务3 |
| 宣传生成 skill | 检索料 + 注入风格样本 + 用户 prompt → 生成富文本 + 附来源/待核实 | 任务2 |

以"技术问答 skill"为例,下面是**实际落地的 `patsnap-tech-qa/SKILL.md` 骨架**(与磁盘一致):

```markdown
---
name: patsnap-tech-qa
description: 当运营/销售/市场用大白话问智慧芽技术问题(产品能力、方法论、技术口径),
  需要有来源、口径统一的标准答案时加载。纯竞品横向对比交给 patsnap-compare;
  写宣传稿交给 patsnap-promo。(Load When 触发条件写在 description 里)
compatibility: 资料由上游检索系统召回,通过 list_materials / read_material /
  check_conflicts 三个工具访问。本 skill 只管"拿到料之后怎么组织成统一口径的答案"。
allowed-tools: [list_materials, read_material, check_conflicts]
---
# 芽懂 · 技术问答(统一口径)
## Goal   口语问题 + 已召回资料 → 挑相关料 → 组织成结论清晰、每个事实绑来源、带时效的答案。
## 先读(Workflow 依赖,必读)  ../references/资料使用SOP.md
## Workflow
  1. 看清有哪些料:list_materials(),按标题/话题/相关度判断哪些相关
  2. 读相关料:read_material(source) 取正文;事实只来自这里,不凭记忆
  3. 处理多来源:check_conflicts();has_conflict 时以 primary 为主答案,
     并把 conflict_note(另一说法+来源+时效)附进回答
  4. 组织答案:先结论后依据,每个事实附来源 URI + 更新时间
  5. 自检:python3 ../scripts/check_sources.py <答案>
## Output Contract  每个事实绑来源;带时间或标"时间未知";无来源标"待核实"
## Boundaries
  - 不编造、不凭记忆;不做检索(料是上游给定的)
  - 冲突以本地权威版为准并标注;单对象问答不做多竞品横向拉表(那是 patsnap-compare)
```

> **架构关键(与 §6 的关系)**:§6 描述的是**接了真 OpenViking 后**的全链路检索 SOP(skill 直接调 `ov search`/`ov grep`)。而**当前落地**把检索侧与生成侧解耦——检索由上游完成(Demo 里用模拟召回,见下节),3 个生成 skill 只通过 `list_materials`/`read_material`/`check_conflicts` 消费"已召回的资料"。两者是同一设计的两个阶段:先把生成侧(挑料/裁决冲突/附来源)做成可跑实物,检索侧待真引擎接入后按 §6 补齐。
>
> **没有一行传统代码**——它用规矩把工具编排成"智慧芽场景下的正确用法"。**这 3 个 SKILL.md 已落地成实物**(见下节"落地状态"),放在 `planA/code/skills/`,Demo/答辩时手上就有可演示的交付物。

### skill 的分层架构(设计原则)

skill 不是随意平铺,而是按"触发准、口径统一、不重复"三条原则组织:

- **做成多个自包含 skill,不做一个大 skill。** OpenViking/Claude 的 skill 都靠 `description` 里的触发条件(Load When)被 agent 加载。具体的触发条件,比一个笼统的大 skill 更准;做成一个大 skill 会被迫在正文里手写意图路由(重造 agent 本就会做的事),且把场景规矩挤进 references 又会引出"怎么保证 references 被读"的矛盾。这也贴合官方示例(`ov-resources`/`ov-add-paper` 都是一 skill 一事)。**当前已落地 3 个生成侧 skill**(问答/对比/宣传);建库规范 skill(`patsnap-ingest`)已在设计中、暂未实现(见下节落地状态)。
- **用料复用:判断内联、产物脚本兜底。** 三个生成 skill(问答/对比/宣传)都要用料。用料里**需要 AI 判断**的步骤(看料→读料→处理多来源→附来源时效)抽成一份共享的 `references/资料使用SOP.md`,各 skill 在 Workflow 里**强制读取**;**机械可验证**的产物(每条结果带来源 + 时效)交给共享脚本 `scripts/check_sources.py` 卡死。**口径不漂移靠脚本兜底,不靠三份文字自觉保持一致**——这正呼应"统一口径"卖点。
- **怎么保证共享规矩被读。** 关键不在"放哪",而在 SKILL.md 正文怎么写。每个 skill 单开一节"**先读(Workflow 依赖,必读)**",写明"未读不得进入下一步"——把读取变成绕不过去的执行动作(与官方 `ov-add-paper` 把"读 references"写成 Workflow 一步同理)。references 不会自动注入,只有写成强制步骤才可靠。
- **纯规矩/清单才共享,场景逻辑不共享。** 用料 SOP 这类**通用规矩**抽进顶层 `references/` 共享(漂移风险低);各场景的判断逻辑留在各自 SKILL.md 正文(触发准、加载一个就完整),如竞品对比的维度对齐逻辑放在 `patsnap-compare/references/对比维度与维度对齐.md`。

### skill 格式选择

以 **Anthropic/Claude 格式为基准 + 兼容 OpenViking 约定**,一份两边都认。理由:执行 skill 的是**外部 Claude Agent**(Demo 后端运行时 / Claude Code),它只调 OpenViking 的 API/CLI 取数据,OpenViking 只是"库"不是执行者——所以 Claude 格式更自洽;`ov skills add` 只是把 skill "托管存放"进 OpenViking,不等于必须用它的格式执行。具体做法:frontmatter 用标准 `name`/`description`/`allowed-tools`/`tags`,正文保留 `Goal`/`Load When`/`Workflow`/`Output Contract`/`Verification`/`Boundaries`/`Runtime Resources`,`description` 里写清触发条件。字段是超集,`ov skills validate` 大概率也能过。

### 落地状态(已实现 / 待补)

**已实现**——3 个生成侧 skill + 共享规矩 + 校验脚本,以及一个能真跑的 Agent 运行时(见第十二节)。实际目录如下:

## 十二、产品 Demo:对话网页承载 skill

只交一份 skill 不够——它是看不见的 markdown 说明书,评委看不见、运营也不会去装。所以要有一个**能演示、运营点得动的产品前台**。但 Demo **不取代 skill,而是承载 skill**:网页是脸,skill 是大脑,OpenViking 是引擎,三者叠起来才是完整交付。

### 形态:一个对话网页,左侧三个能力入口

不是三个独立页面,也不是纯聊天框。是**一个以对话为主的界面**——运营在里面聊天,"技术问答 / 竞品对比 / 宣传生成"是这个对话能干的**三类事**,不是要跳转的三个页面(类比:微信只有一个聊天窗口,但能发文字、语音、转账)。左侧入口只是帮不知道怎么开口的运营**预填提问模板**,本质还是同一个对话。

```
┌─────────────────────────────────────────┐
│  智慧芽技术知识助手                          │
├──────────┬──────────────────────────────┤
│ 快捷入口   │   💬 对话区(主体)              │
│ 💬 技术问答 │   运营:Eureka 多语言比竞品强在哪?│
│ ⚔️ 竞品对比 │   助手:[答案]                   │
│ ✍️ 宣传生成 │        📎 来源 viking://...     │
│  ──────  │        🕐 截至 2026-06          │
│ 📁 知识库   │  ┌─────────────────────────┐ │
│  (原型)   │  │ 输入问题… 或贴竞品网址      │ │
│          │  └─────────────────────────┘ │
└──────────┴──────────────────────────────┘
```

> **为什么不是纯聊天框、也不是五个独立页面**:纯聊天框运营不知道能干嘛,三个可见入口一眼说明能力;五个独立页面要做五套界面还割裂(问完想对比还得切页重来),对话流里一气呵成更像真产品,演示时一个窗口连问三种,节奏不断。

### 三类能力在对话流里怎么走

- **💬 技术问答(主 Demo)**:大白话提问 → 秒回**带来源、带时效**的统一口径答案。演示"统一口径 + 溯源"这个头号价值。
- **⚔️ 竞品对比(爆点)**:甩一个竞品网址 → 几分钟出一张**按维度对齐**的对比表。这是区别于"普通聊天机器人"的差异点。
- **✍️ 宣传生成**:选题 + 一句 prompt → 出**富文本初稿 + 来源列表**。风格由 prompt 主导,不写死模板。

### 关键:后端是"装了 skill 的 Agent",不是写死的代码

网页后端**不把智能写死在代码里**,而是一个**加载了我方 skill 的 Agent 运行时**。这样 skill 才是真大脑、网页只是渲染层——同一个 skill 拆下来装进任何 Agent 都能用,这个 Demo 只是它的一个前台。这也让"skill 是我方核心交付物"这句话立得住。

```
运营发一句话
 → Agent 判断意图(问答 / 对比 / 生成)
 → 加载对应 skill,照 SOP 执行
 → 调 OpenViking(ov find / grep)取料 + 附来源
 → 渲染回对话区(答案 + 来源 URI + 时效)
```

> **设计岔路(必须走对的一条)**:若把查询理解/grep 兜底/对比逻辑直接写死在后端代码里,Demo 也能跑,但"我们交付了一个 skill"就虚了——智能在代码里,skill 成摆设。**正确做法是让智能住在 skill 里,后端只做 Agent 运行时 + 渲染**,否则会被评委一句"那你 skill 到底有啥用"打回去。

### Demo 范围:真做 vs 原型

| 页面/能力 | 做到什么程度 | 为什么 |
|---|---|---|
| 对话区 + 技术问答 | **真做,真跑通** | 主 Demo,统一口径+溯源的核心演示 |
| 竞品对比(含 URL 抓取) | **真做,真跑通** | 差异化爆点,现场冲击力最强 |
| 宣传生成 | 真做,可接受初稿质量 | 官方任务2,链路完整即可 |
| 资料上传页 | 原型/半成品 | 建库是后台动作,不是演示重点 |
| 知识库管理页 | 原型图 | 展示"能管"即可,不必全功能 |

> **安全提醒**:大模型 API key、OpenViking 访问都**走后端代理**,不硬编在前端。竞品抓取只取公开页面,不外传私域内容。现场链路(实时抓取/检索)备一份"预置结果"降级档,答辩不翻车。

> **一句话讲清交付**:"我们交付两样东西——**可复用的能力(skill)**,和**它跑在产品里的样子(对话网页 Demo)**。skill 是大脑、能装进任何 Agent;这个网页是它面向运营销售的一张脸;底下 OpenViking 当引擎。"

## 十三、评估:把"感觉准"变成"数字证明"

怎么证明检索准?需要一个**召回评测集**——既是调优工具,又是答辩证据,还能直接复用到与 Plan B 的对比。

**1. 建 20–30 题召回评测集。** 攒 20–30 个真实运营会问的问题(方法论科普、产品能力、竞品对比各来几个),每题人工标注"理想应召回哪几个资源 URI"。用它量召回率@k、MRR,拿来调 `--uri` 圈定策略、grep 兜底触发条件、top 数量。

**2. OpenViking 自己的评测数据可引用。** OpenViking 公布了知识库问答 benchmark(如 HotpotQA top20 准确率 91%、检索延迟 0.2s 级)。答辩可引用作为"选型依据",但要标注是官方数据、并配上我们自己评测集跑出的结果,不混淆。

### 四方对比(与 Plan B 共用同一张表)

同一批问题,四个参与方各跑一遍。这张表 Plan A 与 Plan B 共用,口径一致才拼得成一张图。

| 参与方 | 是什么 | 这一栏想证明 |
|---|---|---|
| ① 基线 | 通用大模型,无私域知识 | "注入私域知识"有没有价值 |
| ② 普通 RAG | 扁平碎片检索(即 `备选-自研向量库/` 那套) | "结构化组织"是否比扁平切片强 |
| ③ Plan A | OpenViking 成熟开源结构化引擎 | 主推路线的实际效果 |
| ④ Plan B | LLM Wiki 自建编译整页 | 自主可控结构化路线的效果 |

**三层故事**:私域注入有没有用(① vs 其余)→ 结构化组织有没有用(② vs ③④)→ 两条结构化路线 **A vs B 谁实际效果好**(③ vs ④,这是"都做出来看效果"的核心)。

**指标(与 Plan B 共识)**:

| 指标 | 定义 | 打分方法 |
|---|---|---|
| 口径一致性(主) | 同一问题多种问法,答案核心结论是否一致。 | 同题多问法,看结论与来源是否稳定。本方案最该赢的指标。 |
| 事实准确性 | 陈述是否有资料支撑、有无编造。 | 标关键事实点逐条对照,算命中率。 |
| 召回相关性 | 召回的资源是否命中理想 URI。 | 用评测集算召回率@k / MRR。 |
| 溯源完整度 | 关键事实是否都绑了来源与时间。 | 统计带来源 + 更新时间的事实点占比。 |
| 风格贴合度 | 生成内容像不像智慧芽表达。 | 仅评估宣传材料生成任务。 |

> **对比归因要诚实**:③④"各自端到端"做,变量较多。汇报时明确表述"我们对比的是**两种知识组织范式的整体体验**",而非严格控制单一变量的实验。生成/注入层各方尽量保持一致,把变量收窄。用真实数据,样本 5–8 组即可,不要用假数。

## 十四、选型与风险:诚实面对 OpenViking 的边界

直接采用一个成熟开源件,省力也带来依赖。这两个风险必须写进商业企划书的"风险与应对",答辩时主动讲比被问出来强。

**风险① · 开源版默认不是真 BM25。** 中文专名(Eureka/TRIZ/型号)纯向量可能召不回,而这类精确匹配对"技术口径"很关键。真 BM25 混合检索要接入火山商业云 VikingDB(源码 `grep_config.py` 明确注明);开源版默认是本地正则 `grep`。
> 应对:我方编排层判断 query 含专名时并行触发 `ov grep`(它自带的 API)、与语义结果融合(见第六节)。评测集专门放几道专名题验证,也评估是否需要接 VikingDB 商业云换真 BM25。

**风险② · AGPLv3 协议(商业化硬边界)。** 主项目 AGPLv3 有传染性,且比普通 GPL 更严:它的"网络交互条款"意味着即使不分发代码、只通过网络对外提供服务,也可能触发开源义务——这让"HTTP API 黑盒调用、不改源码"这层架构隔离**未必足够**,不能想当然认为架构设计已经解决了合规问题。
> 应对:**商业化前必须由法务正式评估**,而不是"建议咨询"——这是前置硬性条件,不是可选步骤。同时评估火山商业授权路径(CLI `crates/ov_cli`、`examples` 目录已核实为独立 Apache-2.0 许可,较宽松,可作商业化时优先复用的边界)。

**风险③ · 火山生态倾向 & 部署成本。** 原生适配火山系(豆包/VikingDB),Rust+Python 大工程,首次起服务、Ollama 下模型较慢。
> 应对:模型层用兼容接口(可接 OpenAI/内部模型,不绑火山);Demo 用云 API 避开本地模型下载慢;预留半天到一天做环境。

**风险④ · 现场链路崩。** 实时抓取、实时检索在答辩现场可能出意外。
> 应对:主 Demo 备"预置结果"降级档;竞品实时抓取作为可选亮点,失败可回退到预存竞品。

### 组件与分工一览

| 环节 | 谁做 | 说明 |
|---|---|---|
| 文档解析/分层/索引 | OpenViking | 自带,导入即处理 |
| 目录递归检索/rerank | OpenViking | `find` API,可限范围、带来源 |
| 版本/增量更新 | OpenViking | git 快照 + watch |
| 关键件提质 | 我方 + smart-doc | 图多 PPT/扫描件先转 Markdown |
| 查询理解/专名兜底 | 我方编排层 | 改写 + `grep` 融合 |
| 竞品对比/实时抓取 | 我方编排层 | 查询分解 + 维度对齐 |
| 生成/前端/溯源展示 | 我方 | 用户 prompt 主导,附来源 + 时效 |

> **一句话选型**:用 OpenViking 当引擎,把一周的自研省下来投到贴合场景的补强上。它替我们做掉解析、分层、检索、版本;我方补专名召回、竞品对比、前端与溯源展示。清醒面对 BM25 缺失和 AGPL 边界这两条,方案就站得住。
