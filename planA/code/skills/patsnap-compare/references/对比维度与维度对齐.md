# 竞品对比：查询分解与维度对齐

> 竞品对比是"芽懂"的差异化爆点。核心是**查询分解 + 我方/竞品分别用料 + 按维度对齐**。
> 把两边的料混在一起答会偏，所以必须分解、分料、再对齐。
> 前提：资料（我方产品料 + 竞品料）已由上游检索召回放进本次 case，本 skill 只用不检索、不抓网页。

## 一、为什么要分解

用户问"我们 Eureka 多语言比竞品X强在哪"，若把召回的料混在一起答，结果会混、会偏。
必须拆成：{对比维度、我方对象、竞品对象、时间意图}，我方料一路、竞品料一路分别读，再逐维度对齐。

## 二、查询分解（把口语转结构化）

从问题中抽出：

```yaml
is_comparison: true
dimensions: [多语言, 检索精度, ...]   # 对比的维度；用户没说全时按能力维度补默认
our_target: eureka                    # 我方对象
competitors: [竞品X]                  # 竞品对象
time_intent: latest                   # 默认取最新
```

## 三、从已召回资料里区分我方料与竞品料

先 `list_materials()` 看全部元信息，按 `source` 和 `topic` 分两堆：

- **我方料**：`source` 含 `products/`（如 `.../products/eureka/lang-support.md`），或 `topic` 指向我方对象（如 `eureka-lang`）。
- **竞品料**：`source` 含 `competitors/`（如 `.../competitors/竞品X/lang.md`），或 `topic` 指向竞品对象（如 `competitor-x-lang`）。

分好堆后，对相关的料分别 `read_material(source)` 取正文。**事实只能来自读到的料，不凭记忆。**

## 四、处理同话题多来源

调 `check_conflicts()`。同一 `topic` 下有多份料且说法不同时（例如我方产品有新旧两版、语言数不一致）：

- 以 `primary`（更权威/更新）为主答案填进对应格子；
- **必须**把 `conflict_note`（另一说法及其来源、时效）附进该格——曝光而非静默丢弃。
- 这是"统一口径"在对比场景的体现：告诉用户"以哪个为准、还有哪个不同说法"。

## 五、按维度对齐

逐个维度，把我方与竞品的信息摆到同一行，形成对比表：

| 维度 | 我方(来源/时间) | 竞品X(来源/时间) |
|---|---|---|
| 界面语言 | 支持 12 种(products/eureka/lang-support.md, 2026-06-20) | 支持 6 种(competitors/竞品X/lang.md, 2026-04-01) |
| 跨语言语义检索 | 支持(同上) | 不支持(同上) |

- **缺失维度明确标"未找到"**，不猜、不脑补。
- 每格都带来源 `source` + `updated_at`；缺 `updated_at` 标"时间未知"。

## 六、竞品维度资料不足怎么办

新架构下"去哪找料、找得全不全"归上游检索侧，本 skill 不自己抓网页。
若某竞品的某个维度在已召回资料里没有：

1. 该格明确标 **"未在已召回资料中找到该竞品的X维度信息"**，绝不脑补一个数字或结论。
2. 提示：**这需要上游检索补充该竞品的 X 维度资料**（这是对检索侧的补料诉求，不是本 skill 的活）。
3. 其余能对齐的维度照常出表，不因一格缺失而中断整张对比。

> 为什么这样：找料、清洗、结构化本属检索侧职责；生成侧若自己去外部取料，会引入不可控的脏数据、绕过统一的溯源与冲突裁决。把缺口清晰暴露给上游，比自己硬填更可靠。
