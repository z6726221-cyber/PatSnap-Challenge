把技术资料沉淀成一个能问、可溯源、口径统一的知识库。

本方案是课题2的主方案(Plan A)。产品面向运营、销售、市场、内容策划——他们不懂技术、又天天要在客户面前讲技术。系统把智慧芽的技术资料沉淀为可检索知识库,让他们想问什么问什么,并解决最痛的问题:技术口径不一致。核心不在生成花样(回答风格、报告样式由用户自己写 prompt 控制),而在数据怎么存、怎么召回,以及内外数据如何结合、口径如何随时间保持一致。

一、产品定位:先想清楚给谁用

用户是运营销售,不是工程师

技术选择全都由"用户是谁"决定。这个产品的用户不懂技术,他们的诉求和工程师完全不同,方案的每个取舍都要顺着这一点。

用户:运营 / 销售 / 市场 / 内容策划

他们不懂 TRIZ、不懂专利技术细节,平常遇到技术问题都去问研发,研发又讲不明白、响应慢。他们要的是用大白话问、拿到能直接用的答案。

最痛的点:技术口径不一致

不同销售对同一个技术能力说法不同,在客户面前容易露馅。知识库的头号价值是统一口径——同一个问题,谁问都是同一个有出处、有时效的标准答案。

核心是"存"与"召回"

回答风格、生成什么样的报告,都由用户自己写 prompt 决定。所以本方案不在生成花样上堆工程,重心是数据怎么存、怎么精准召回。

溯源是刚需

"统一口径"要经得起客户追问。每个答案必须带来源,让销售知道这话是哪份材料、哪个版本说的,而不是模型编的。

内外数据结合

宣传智慧芽必然有横向对比。系统要能结合本地能力 + 外部竞品信息——需要时自己联网搜,或用户抛一个竞品网址,让系统说清我方优势。

范围:做透 1-2 个项目 + 五个知识模块。深度上选 1-2 个开源项目(如 Eureka)把产品能力讲透;广度上靠五个模块(方法论/产品能力/宣传风格语料/竞品信息/技术路线)把运营会问的面铺全。不贪多,流程完整比功能多重要。

二、整体架构

五段链路

系统是一条五段管线。与通用 RAG 相比,本方案把工程重点压在分片质量、检索精度、时间线一致、内外结合四处,而不是生成端。

① 数据接入→ ② 分片 + 建库→ ③ 四阶段检索(本地+外部)→ ④ 生成(用户 prompt 主导)→ ⑤ 溯源 + 时效标注
各段对应章节:① → 三;② → 四、五;③ → 六(含七时间线、八内外结合);④⑤ → 九评估。第十节讲工程部署与选型。生成端刻意做薄——它由用户 prompt 驱动,不是本方案的差异点。

三、数据接入

Word / PDF / PPT / 代码,各有处理

四类数据源用两条通道处理:文档类(Word/PDF/PPT)统一走 Smart-Doc,代码类单独处理。目标是把异构文件都转成统一的内容块,供后续分片。

1

Word / PDF · Smart-Doc API

调 /pdf/smart-doc(PDF)和 /document/smart-doc(Word,内部先转 PDF 再处理)。参数用推荐默认:detailed:true, markdown:true, ocr_backend:"auto"。

先讲清它返回什么:Smart-Doc 做版面分析 + OCR + Markdown 聚合,产出拼接好的 Markdown 文本和带 bounding box 的版面块。它不直接给"标题树"或"表格行列 JSON"——标题层级和表格要从它输出的 Markdown 里解析(#/## 和表格语法已够还原)。这一步只写一个适配器:Markdown → 统一内容块。

2

PPT · 同样走 Smart-Doc,但要额外处理两点

/document/smart-doc 明确支持 ppt/pptx(先转 PDF 再走版面分析+OCR),不用单独造轮子。但 PPT 有两个坑必须处理:

① 标题页/过渡页(只有大标题没内容)要并入下一页,否则产生一堆没信息的空块。② 图表页(架构图、竞品对比图)转 PDF 后文字很少、信息在图里——对这类页把 Smart-Doc 返回的图片块单独送一次多模态模型,生成一段描述拼进正文。否则问"你们架构长啥样"根本召不回。

3

代码库 · 文档层为主,函数层为辅

关键判断:用户是运营,几乎不问函数实现。所以代码库重心不在索引每一行代码,而在"讲清它能干什么":

README、设计文档、模块说明 → 走文档切法(主力,运营真正会用到)。代码函数 → tree-sitter 按"函数签名 + docstring"一块、函数体丢弃(轻做,备查)。别在函数级索引上投重工程。

力气分配:解析用官方 Smart-Doc、不自研 OCR。真正决定质量的是下一步的分片和元数据,那才是要人理解、不能偷懒的地方。

四、分片:决定召回质量的七成

分类型切 · 表格线性化 · 上下文前缀

分片的核心矛盾:块太大,一块混几个意思、向量被"平均"掉,召回不准;块太小,丢上下文、召回到一句孤话没法用。目标是一块 = 一个完整自洽的意思。不能全库一个切法。

数据类型切法块大小特殊处理
Word / PDF按标题层级 → 段落300–600 字保留标题前缀、约 50 字重叠
PPT按页,一页一块一页空标题页并入下页;图表页补图片描述
代码-文档Markdown 段落300–600 字主力
代码-函数签名 + docstring一函数一块丢函数体,轻做
风格样本整篇不切整篇单独存,权重低

难点1 · 表格:你的高频数据,naive 切法必废

你的资料里全是对比表(功能对比、参数表、竞品矩阵),这恰是分片最容易出事处。把表当普通 Markdown 塞进去,销售问"我们多语言比竞品强在哪",向量很可能召不回——因为"25 种/10 种"这些数字在向量空间里没有语义信号。

解法:表格行级线性化。把每一行连同表头改写成一句自然语言、各自成块:
• "在『多语言支持』维度,智慧芽 Eureka 支持 25 种,竞品A 支持 10 种,竞品B 支持 8 种。"
• "在『实时更新』维度,智慧芽 Eureka 支持,竞品A 不支持,竞品B 支持。"
这样每个维度都能被独立、精准召回。小表(几行)可整表存一块并加一句"本表对比 X/Y/Z 三家在……的差异";大表逐行线性化。此项对本产品价值极高——竞品对比是核心场景。

难点2 · 混主题文档:靠标题层级救

一份白皮书常把"产品能力+定价+客户案例"混在一起。只要 Smart-Doc 还原出标题层级,按标题切就能自然分开。万一原文是没标题的大段流水,数据量不大(1-2 个项目),人工补几个分隔标题比上自动语义分割靠谱得多,别为这个上重工程。

难点3 · 上下文前缀:不是魔法,是拼装

给每块拼一段"出身信息"再 embedding,是提升召回的低成本高回报手段,务必做。它就是已有的元数据拼出来的,不用额外 AI:

[项目:Eureka | 模块:产品能力 | 文档:产品白皮书v2 | 章节:多语言支持] Eureka 目前支持 25 种语言的专利检索……(块正文)

项目/模块 来自建库时打的标签,文档/章节标题来自 Smart-Doc 还原的标题层级。好处:①只写"支持 25 种"的短块,靠前缀知道自己在讲什么;②标题、专名进了向量,专名召回率上升。存储时正文和前缀都留着,展示给用户时只显示正文。只有一种情况需额外调一次小模型:块本身极短又没上下文(如线性化后只剩"支持 25 种语言")时,让模型补一行摘要。

五、知识库构建:不是"一个向量库",是四件套并存

向量 + BM25 + 元数据 + docstore

一个能用的知识库,底下是三个索引 + 一份原文。只建一个向量库,是很多人后来抱怨"检索不准"的根因。

① 向量索引

存每块 embedding,负责语义相似召回。选型关键:用中文/多语言强 embedding 模型(bge-large-zh、m3e,或内部中文向量 API,约 1024 维)。用默认英文模型处理中文专业文本会很差。不建议为挑战赛做微调——数据小、收益边际、不划算,选对基础模型比微调重要。

② 关键词索引(BM25 / 倒排)

负责专名精确匹配。中文里 Eureka、TRIZ、版本号这种,纯向量经常召不回,必须它兜底。中文坑:BM25 必须先分词(jieba 等),否则把"多语言支持"当一整个 token 匹配不上、几乎失效。分词时把 Eureka/TRIZ/产品名加进自定义词典,别被切碎。

③ 元数据索引(结构化字段)

存 source_type / project / module / version / is_latest / updated_at,负责先圈范围、按时间线过滤。检索第一步不是撒向量,是用它把范围锁死。

④ 原文档存储(docstore)

按 parent_doc_id 存整篇原文。召回到某块后,常需把它所在章节的前后文一起取出喂模型(只给孤块不够)。这叫"小块检索、大块喂给模型"。

每块的数据模型(竞品与本地同一套 schema,仅 source_type 区分,便于检索时分开取):

{ "chunk_id": "eureka-whitepaper-v2-0012", "parent_doc_id": "eureka-whitepaper-v2", "source_type": "official | paper | code | doc | ppt | competitor", "project": "eureka", "module": "产品能力", // 五模块之一 "competitor": null, // source_type=competitor 时填竞品名 "title": "多语言支持", "version": "v2", // ← 时间线:哪个版本的资料 "updated_at": "2026-05", // ← 更新时间,召回时优先新的 "is_latest": true, // ← 同主题多版本时标记当前有效版 "supersedes": "eureka-whitepaper-v1", // ← 指向它替代的旧版 "source_url": "https://...", // ← 溯源:统一口径的可信度前提 "context_prefix": "[项目:Eureka | 模块:产品能力 | ...]", "content": "……", "embedding": [/* 1024 维 */] }
建库流程:原始文件 → Smart-Doc / tree-sitter 解析成统一内容块 → 分片(按类型规则)→ 每块打元数据 + 拼上下文前缀 → embedding → 同时写入 向量索引 + BM25 索引 + 元数据索引 + docstore。

六、检索:四阶段流水线

不是"查一次向量",是一条流水线

运营问得口语、模糊("这玩意儿比别人强在哪"),直接查向量质量会很差。精准检索是一条四阶段流水线。

0

阶段0 · 查询理解(最被低估的一步)

原始 query 先过一次小模型调用,转成结构化条件再检索:识别意图/范围(涉及哪个项目、哪个模块?问事实、要对比、还是要最新?)→ 转成元数据过滤条件;查询改写(把"这玩意儿"补成"Eureka 的核心能力",生成 2–3 个变体);时间意图("最新/目前"→ 强制 is_latest)。范围圈对了,后面就对了一大半。

1

阶段1 · 混合召回(向量 ∥ BM25)

在阶段0圈定的元数据范围内,同时跑向量 kNN 和 BM25,各取 topN,用 RRF(倒数排名融合)合并。向量管语义、BM25 管专名,两条腿走路。收益主要来自"范围圈得对"和"两路互补",不是权重调参,不要过度调权重。

2

阶段2 · 重排(Rerank)

合并后的候选(20–30 条)过一个 rerank 模型,按与 query 的真实相关度重新打分,取 top 5–8。这一步对最终精度提升最明显,是向量方案能"秀工程"的点。

3

阶段3 · 上下文组装

拿到 top 块后不直接塞:按 parent_doc_id 补齐同文档相邻块(小块检索、大块喂);按 updated_at 择新去重,同主题旧版丢掉;每块带上 source_url + updated_at,为溯源和"口径时效戳"做准备。

口语 query→ 0 查询理解→ 1 元数据过滤 → 向量∥BM25 → RRF→ 2 rerank → top5-8→ 3 补相邻块+择新+附来源→ 喂大模型

最难的场景 · 对比类提问

你的销售天天问对比类问题:"我们 Eureka 的多语言支持比竞品X强在哪?"这种 query 用单次召回会出事——会偏向召回一边(要么全本地、要么全竞品),对比不起来。解法是查询分解 + 多源分别召回 + 按维度对齐:

用户问:"Eureka 多语言支持 vs 竞品A 强在哪?" 阶段0 查询理解 → 结构化: { "is_comparison": true, "dimension": "多语言支持", "our_target": {"project":"eureka"}, "competitors": ["竞品A"], "time_intent": "latest" } 阶段1 分两路分别召回(元数据过滤不同): 路A: filter{project=eureka} → 本地"多语言"相关块 路B: filter{source_type=competitor, competitor=竞品A} → 竞品块 (竞品库没有 → 触发实时抓取,见第八节) 阶段3 按"多语言"维度把两边对齐,一起喂模型
没有这个分解步骤,对比类回答质量一定差。而这恰是本产品的核心卖点(内外数据结合),所以这块工程必须做实,不能当普通召回处理。

七、时间线 / 版本:让口径不"越用越乱"

文档级版本 + 只召最新

同一主题资料会有新旧两说(旧 PPT 写"支持 10 种语言",半年后新材料写"25 种")。两条都被召回,模型就会说错或自相矛盾——这就是"逻辑乱"。

版本以"文档"为单位(推荐)

同一份资料的 v1/v2 整份替换,老版标 is_latest=false 归档、不进主检索库。谁是谁的新版靠 parent_doc_id + 人工确认。数据量不大,人工就能标清,干净可靠、Demo 完全够。

不做"主题/事实级"版本(不推荐)

给每条事实抽主题、跨文档比对新旧,需要实体抽取+主题聚类,复杂且易错,挑战赛时间不划算。明确不做,是取舍不是遗漏。

召回层三条规则:

  • 默认只召回 is_latest=true,旧版归档库不参与,从源头避免新旧混召。
  • query 带"最新/目前/现在"时强制走最新;带"历史/以前版本"时才放开归档库。
  • 兜底:万一同主题多版本仍被召回,生成前按 updated_at 择新,并在输出标注"截至 2026-05 的口径"。
与"统一口径"的联动(答辩亮点):输出除了带来源,还带资料更新时间。销售看到的是"答案 + 出处 + 这条口径截至几月"——口径本身带了时效戳,就不会拿半年前的旧说法去客户面前讲。统一口径 = 统一内容 + 统一时效。

八、内外数据结合

预存竞品(主 Demo)+ 实时抓取(亮点)

销售宣传必有横向对比,外部竞品数据是刚需。采两条腿:主 Demo 用预存竞品资料保稳,另做"抛竞品网址实时抓取"作亮点。

预存竞品(主 Demo,稳)

提前把几个主要竞品资料抓好、结构化、存进库(source_type=competitor)。现场不联网,稳定可控。竞品库与本地同一 schema,检索时用元数据把"本地能力"和"竞品能力"分开取。

实时抓取竞品 URL(亮点)

用户抛一个竞品网址,系统抓取该页内容 → 清洗 → 按对比维度对齐 → 与本地能力逐维度比。答辩演示最有冲击力,直击"销售天天被这样搞"的痛点。

抓来的网页内容不可信、格式乱,不能让模型自由发挥。流程要结构化:抓取 → 去噪(去导航/广告/无关块)→ 让模型按"我方已知的对比维度"逐项抽取竞品对应信息(维度由本地能力驱动,不是漫无目的地读)→ 逐维度对齐生成对比。缺失维度明确标"未在该页找到",不猜。
外部数据一视同仁落溯源:抓来的内容同样带 source_url + 抓取时间进溯源。冲突时以本地权威为准并标注。联网只取公开信息,不外传任何私域内容。

九、评估:把"感觉准"变成"数字证明"

召回评测集 + 三方对比

怎么知道检索到底准不准?靠感觉调参是调不出来的。需要一个召回评测集——它既是调优工具,又是答辩证据,还能直接复用到与 Plan B 的对比。

1

建 20–30 题召回评测集

攒 20–30 个真实运营会问的问题(TRIZ 科普、产品能力、竞品对比各来几个),每题人工标注"理想情况下应该召回哪几个块"。

2

量召回率@k 和 MRR

用这个集合量 召回率@k、MRR,拿它去调分片大小、混合检索权重、rerank。从"感觉还行"变成"有数字证明检索准"。答辩时直接甩召回率曲线,回应"你怎么证明检索好"。

三方对比 · Plan A vs Plan B vs 基线

同一批问题,用三种方式各跑一遍:通用大模型 / Plan A 向量方案 / Plan B LLM Wiki,按指标打分。

指标定义打分方法
召回相关性召回的块是否命中理想块。用评测集算召回率@k / MRR。
事实准确性陈述是否有资料支撑、有无编造。标出关键事实点逐条对照,算命中率。
口径一致性同一问题多次/多人问,答案与时效是否一致。同题重复问,看答案与来源是否稳定。
溯源完整度关键事实是否都绑了来源与时间。统计带 source_url + updated_at 的事实点占比。
对比归因要诚实:Plan A 与 Plan B "各自端到端"做,变量较多。汇报时明确表述"我们对比的是两种知识组织范式的整体体验",而非严格控制单一变量的实验;并在事实准确、召回相关这类指标上尽量让两边的生成/注入层保持一致,把变量收窄。用真实跑出的数据,样本 5–8 组即可,不要用漂亮的假数。

十、工程与选型

组件 · OpenViking · 风险

组件

组件清单

  • 解析:官方 Smart-Doc API(结果本地缓存,同文档不重复解析)。
  • 向量库 / 检索:支持"元数据过滤 + 向量检索"的库;万级切片以内单节点足够。
  • 关键词:BM25(带中文分词 + 专名词典)。
  • 向量化 / rerank:中文强 embedding + rerank 模型,离线批量跑,不常驻 GPU。
  • 生成:主流大模型 API(回答风格由用户 prompt 主导)。

与产品原型的关系

  • 本方案是产品原型里"智能问答页/竞品分析页"的引擎。
  • 主 Demo 现场真跑通:口语提问 → 四阶段检索 → 带来源+时效的答案。
  • 竞品分析页展示预存对比 + "抛 URL 实时抓"亮点。
  • 知识库样例就是可交付的知识库结构 + 元数据。

向量后端选型 · OpenViking(待补)

⏳ 待定,等 OpenViking 资料补齐。本方案先按"通用向量库"写。拿到 OpenViking 的 README/文档后,在此评估它对建库四件套(向量索引 / BM25 / 元数据过滤 / docstore)各覆盖了几样、缺的要不要自己补,并给出"直接用 / 借鉴思想 / 换轻量库"的选型判断。

初步倾向:向量后端只填"① 向量索引(可能带 ③ 元数据过滤)"这一格,不触碰分片、检索流水线、时间线、内外结合等上层逻辑——所以它只能让一层更快更省,不改变方案骨架。"借鉴思想"通常比"直接绑定一整套外部栈"更稳,尤其本方案主题是私域数据、要避免过重依赖与数据治理疑问。

主要风险与应对

风险

  • PPT 图表页 OCR 后信息丢失。
  • 实时抓取的竞品网页格式乱、不可信。
  • 资料多版本导致口径冲突。
  • 现场实时链路崩。

应对

  • 关键图表页补多模态图片描述。
  • 抓取走"去噪→按维度抽取→缺失标注"的结构化流程,不放养。
  • 文档级版本 + 只召最新 + 时效戳标注。
  • 主 Demo 备"预置结果"降级档;先跑通基础检索再加 rerank。
一句话:本方案把工程压在分片(表格线性化)、四件套建库、四阶段检索(对比类分解)、时间线一致、内外结合五处——这些才是"给运营销售统一技术口径"这件事的真正难点。生成端刻意做薄,因为它由用户 prompt 驱动。
